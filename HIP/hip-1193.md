## Abstract

This HIP defines the requirements and implementation details for transitioning the Hedera network
from record and event streams to block streams. This transition is a critical step toward enhancing
blockchain compatibility, improving data integrity, and enabling future network optimizations such
as state proofs.

The transition is structured as a **two-stage cutover**:

- **Stage 1 — WRB Cutover**: Consensus nodes begin producing Wrapped Record Blocks (WRBs) and
  streaming them to a limited set of block nodes for integration testing and infrastructure
  validation. Record streams continue to be produced and remain the cryptographic authority of the
  network during this stage. Mirror nodes may optionally be configured to receive WRBs, but will
  automatically fall back to record streams if WRBs cannot be received or validated.

- **Stage 2 — TSS Cutover**: The full cryptographic cutover. Record and event stream production
  ceases, TSS-signed block streams become the authoritative data source, and mirror nodes
  permanently transition to block stream processing.

The proposal outlines the coordinated changes required across consensus nodes, mirror nodes, and
supporting infrastructure to ensure a clean cutover without service disruption. It specifies new
buckets, storage paths, file formats, and mechanisms to handle the transition.

## Motivation

Following the approval of HIP-1056, HIP-1081, and HIP-1200, the data output produced by the Hiero
network will undergo a significant upgrade. This upgrade modernizes the network's cryptography,
standardizes and optimizes data formats, and introduces block nodes to provide greater network
resilience and improved data availability. The key enhancements offered by this change include:

- **Unified Data Stream** — The block stream consolidates event streams, record streams, sidecars,
  and signature files into a single cohesive data stream.
- **State Data** — Each block includes the complete set of state change data for its consensus
  round, enabling block nodes to maintain state in parallel with consensus nodes.
- **Verifiable Proofs** — Each block is independently verifiable and contains full cryptographic
  proofs of all transactions, execution trace data, network consensus, and resulting state changes.
  Block nodes additionally provide dedicated state proof services.
- **Comprehensive Protobuf Specification** — By defining blocks using Protocol Buffers, the inputs,
  outputs, and state changes produced by consensus nodes are explicitly and unambiguously specified.
  This lays a strong foundation for future implementations and promotes greater client diversity.
- **Network Scalability** - Block nodes assume responsibility for state-related services—including
  node reconnection (re-sync), state proof generation, and state backups—thereby relieving consensus
  nodes of these duties and allowing them to dedicate their full resources to transaction processing
  and consensus.
- **Reduced Transition Risk** — A two-stage cutover allows the block stream infrastructure to be
  validated under real network conditions before the irreversible cryptographic cutover to TSS
  occurs, significantly reducing the risk of data loss or service disruption.

Given the scale and interconnected nature of these changes, a structured migration plan is essential
to avoid disruption for mirror nodes and other downstream consumers that currently depend on legacy
record streams.

## Rationale

The transition from record streams to block streams requires careful planning to ensure network
continuity, data integrity, and backward compatibility. Block stream data will be available both
from block nodes and limited history cloud buckets. This HIP proposes a clean cutover approach with
the following key design decisions:

### Two-Stage Cutover for Risk Mitigation

The cutover is divided into two discrete stages to minimize risk. The first stage relies on a modified form of the familiar record file data that we call a Wrapped Recordfile Block (WRB). The second stage completes the transition with cryptographic upgrades with the Threshold Signature Scheme (TSS).

**Stage 1 (WRB Cutover)** serves as a live integration test of the block stream infrastructure.
Consensus nodes produce WRBs in parallel with record streams. Record streams remain the
authoritative cryptographic source — if bugs or issues are discovered in WRB production or
streaming, consensus nodes continue producing record files unaffected. This stage allows block
nodes, mirror nodes, and downstream consumers to validate the new format and delivery pipeline
without any cryptographic or operational commitment.

**Stage 2 (TSS Cutover)** is the full, irreversible cryptographic cutover. This stage only
proceeds once Stage 1 has met defined readiness criteria, including successful WRB streaming
to block nodes, successful mirror node ingestion of WRBs, and verified block proof correctness.
During this time we will also stress-test the TSS implementation and ensure that the new
cryptography and signature processes function correctly across all functionality and
performance ranges. At this point, record and event stream production ceases and TSS-signed
block streams become the sole authoritative source of network data.

This staged approach separates the infrastructure risk (can we produce and deliver block streams
correctly?) from the cryptographic risk (can we permanently commit to TSS-signed blocks?), allowing
each to be addressed independently.

### Seamless Mirror Node Transition

The mirror node will automatically detect when the Stage 2 - TSS cutover occurred by periodically checking for
block stream data if no record streams have been received in a measurable amount of time. If no
block node connection information is available either via manual configuration or HIP-1137
Registered Node transactions, the mirror node will fall back to bucket-based retrieval
automatically. If it detects block stream data in either a configured block node or cloud storage,
it will permanently transition to using block streams. This approach:

- Allows mirror nodes to detect the Stage 2 transition without configuration changes.
- Prevents data loss during the transition.
- Eases mirror node operations, enabling mirror nodes to update the mirror node version before the
  block stream cutover and have seamless service continuity.

During Stage 1, mirror nodes that are configured to receive WRBs will automatically fall back to
record streams if WRBs cannot be received or validated. This fallback is automatic and requires no
operator intervention.

### Dual Publishing and Cloud Storage

Block streams are accessed by mirror nodes from both block nodes and cloud storage buckets, to
provide redundancy and high availability. Block nodes are the primary delivery mechanism, with
cloud storage serving as a fallback if a block node connection is unavailable or fails. Mirror
nodes will still require a software upgrade to support the block stream format regardless of which
source they retrieve data from.

New cloud storage buckets will be created to distribute this block stream data. These buckets are
limited-retention and contain only the most recent data. Syncing data from genesis will require
connecting to block nodes or importing a state snapshot. New buckets are created rather than
reusing existing ones to allow retention policies to efficiently prune historical data.

Mirror nodes configured for Hedera networks will automatically use the new buckets with no
configuration required. Operators only need to ensure their existing cloud storage credentials
have the necessary permissions to read the new buckets before the cutover.

### Resettable and Ephemeral Networks

The new block stream files will follow a revised path structure that includes a network identifier.
The network identifier will allow for additional flexibility in non-mainnet environments including
the ability to provide ephemeral networks, multi-network buckets, and automatic resets. Having the
network name included in the path allows for the option of having data from multiple networks
stored in the same bucket. This flexibility will reduce the maintenance burden on the operations
team.

### Cost Efficient Block Retrieval

When connected directly to a block node, blocks will be directly read by consumers like mirror
nodes via a streaming gRPC API as defined in HIP-1081. The block streams uploaded to cloud storage
will be compressed and named for efficient retrieval. Block files will use the `.blk.zstd`
extension and Zstandard compression to provide efficient data compression, and reduce bandwidth
and storage costs.

The file name will be the block number instead of a date as used in the record stream. This will
allow mirror nodes to directly download the next sequential block number and avoid costly list
operations. The 19-digit block number will be partitioned into 4-digit paths to reduce the amount
of data that matches a single prefix in cloud storage. This will allow for efficient listing of
folders to aid in debugging or pruning.

### Unified Data Format

A key design goal of this transition is to achieve a unified block stream format across the entire
ledger history, not just from the cutover point forward. To accomplish this, legacy record files,
sidecars, and signature files will be reprocessed into Wrapped Record Blocks (WRBs), ensuring that
historical data is accessible and verifiable within the same block-stream architecture as new data.
This means that consumers of network data — including mirror nodes and block nodes — will be able
to traverse the full history of the ledger through a single standardized format.

## User stories

1. As a mirror node operator, I want a means to automatically transition from record stream
   processing to block stream processing so that I don't miss any transactions during the cutover
   or cause downtime in my application.

2. As a mirror node operator, I want to understand the new block file format and paths so that I
   can adjust my infrastructure to process block streams efficiently.

3. As a developer building on Hedera, I want to understand how the transition affects data
   availability and processing so that my applications continue to function correctly.

4. As a consensus node operator, I want to know exactly what changes are needed to support block
   streams so that I can prepare for the upgrade.

5. As a mirror node operator, I want to know what version of the mirror node software I need to be
   running and have enough time to upgrade to that version before the cutover.

6. As a network user, I want the transition to be seamless with no impact on transaction processing
   or data availability.

7. As a consumer of mirror node data and associated metrics, I want the transition to be seamless
   with no impact on how I consume mirror node data.

8. As a mirror node operator, I want discovery of registered block nodes and my configuration
   updated automatically to subscribe to them.

9. As a mirror node operator participating in Stage 1 testing, I want my node to automatically
   fall back to record streams if WRBs cannot be received or validated, so that I have no risk of
   data loss while participating in the WRB integration test.

10. As a mirror node operator, I want the option to opt in to receiving WRBs during Stage 1 so
    that I can validate my block stream integration before the Stage 2 cutover.

## Specification

### 1\. Ledger ID Generation

To prepare for the transaction, some work will take place in the release prior to the cutover:

- Consensus nodes will perform the TSS ceremony [HIP-1398](https://hips.hedera.com/hip/hip-1398).
- The ledger ID will be generated and externalized by the consensus nodes via a synthetic
  `LedgerIdPublicationTransaction` as defined in HIP-1200.
- Mirror nodes will receive this transaction and persist the ledger ID in their respective
  databases.
- Mirror node code will be updated to hardcode this ledger ID per network for any operators that
  start after this ledger ID publication.

### 2\. Cutover Point Definition

The transition from record streams to block streams occurs in two discrete stages.

#### 2.1 Stage 1 — WRB Cutover

The WRB Cutover is a live integration test of the block stream pipeline. The following conditions
define this stage:

- Consensus nodes begin producing WRBs alongside continuing record stream production.
- WRBs are streamed to a limited, designated set of block nodes for integration testing.
- **Record streams continue to be produced and remain the cryptographic authority of the network.**
  Consensus nodes will not halt record file production due to bugs or issues with WRB streaming.
- Mirror nodes may be optionally configured to receive WRBs during this period.
- Mirror nodes configured to receive WRBs will automatically revert to record streams if WRBs
  cannot be received or fail validation.
- This stage does not constitute a cryptographic cutover. No TSS signing of blocks occurs during this stage.

Stage 1 concludes and Stage 2 is authorized only when all of the following readiness criteria
are met:

- WRBs are successfully streamed to and stored by designated block nodes without errors.
- At least one mirror node has successfully ingested and validated WRBs end-to-end.
- Block proof correctness has been verified for a sustained period.
- No critical defects remain open against WRB production or delivery.
- TSS and WRAPS processing is fully tested and validated
- TSS and WRAPS meets all relevant performance and correctness criteria.

#### 2.2 Stage 2 — TSS Cutover

The TSS Cutover is the full cryptographic cutover from record streams to block streams. This stage
proceeds only after Stage 1 readiness criteria are satisfied, and occurs at a network update
boundary:

- Occurs after a Freeze Upgrade transaction during the mainnet update window before transition.
- The current round comes to consensus.
- The final record stream will be completed.
- WRB generated from the final record in the record stream.
- First TSS-signed block of block stream produced.

### 3\. Consensus Node

#### 3.1 Wrapped Record Blocks for Historical Data

To allow migration to a single standardized data format while fully preserving data integrity and
cryptographic verifiability, historical data — including legacy record files, sidecars, and their
associated signature files — will be wrapped into Wrapped Record Blocks (WRBs) during the
transition period.

Each WRB is constructed by:

* Embedding the original record stream contents within a Block Item.
* Appending the required block header (including consensus round metadata and timestamps) and
  footer.
* Generating a cryptographic block proof derived from the individual node RSA signatures
  originally produced for that record file.

This operation ensures that all historical data remains accessible, verifiable, and compatible with
the new block stream architecture without loss of provenance or trust assumptions. While the
original record files, sidecars, and signatures will continue to be preserved in publicly
accessible cold storage, block nodes will maintain a complete historical archive of WRBs to
provide efficient, unified access to past network data within the block stream format.

> **Note:** Existing mirror nodes will not be affected by the introduction of WRBs in the block
> stream.

#### 3.2 Stage 1 — WRB Streaming Behavior

During Stage 1, consensus nodes will:

- Begin producing WRBs for each record file and streaming them to the designated Stage 1 block
  nodes.
- **Continue producing record and event streams without interruption**, regardless of the status
  of WRB production or streaming. Record stream production is not conditional on WRB success.
- Not upload WRB based block streams to cloud storage buckets during Stage 1. Cloud bucket publishing
  begins at Stage 2.
- Not apply TSS signing to blocks during Stage 1. RSA-derived block proofs (as described in
  Section 3.1) are used.

#### 3.3 Stage 2 — Record and Event Stream Termination

At Stage 2, consensus nodes will:

- Perform a Freeze operation and stop accepting transactions.
- Freeze round comes to consensus.
- Ensure all signatures are collected and files are properly finalized.
- Produce the final record file.
- Stop producing both record and event streams at the designated cutover point.
- Shutdown uploaders as uploading responsibility is moved to block nodes.
- Process the final record file into a Wrapped Record Block (WRB). The root hash of this WRB will
  be included as the Previous Block Hash in the first block produced after the cutover, ensuring
  cryptographic continuity across the transition.

#### 3.4 Stage 2 — Block Stream Initialization

At Stage 2, consensus nodes will:

- Construct the first block.
- Apply the correct block number from the final record.
- Add the root hash of the last record WRB as the Previous Block Hash.
- Sign the block using TSS and the ledger ID previously generated.
- Push block streams to the configured block nodes.

### 4\. Block Node

#### 4.1 Stage 1

During Stage 1, a limited set of designated block nodes will:

- Accept WRB streams from consensus nodes.
- Store WRBs for mirror node consumption by participating operators.
- Provide streaming access to WRBs for mirror nodes configured to receive them.
- Report ingestion metrics and errors to support readiness gate evaluation.

Block nodes not designated for Stage 1 testing are not required to participate until Stage 2.

#### 4.2 Stage 2

At Stage 2, all block nodes will:

- Upload block streams to the new buckets following the new path structure.
- Deduplicate block streams by block number so only one copy is stored in the bucket.
- Provide immediate streaming access to exported consensus network data.

### 5\. Cloud Storage

#### 5.1 Bucket Creation

The following are the three new buckets:

- Mainnet: `hedera-mainnet-recent-block-streams`
- Testnet: `hedera-testnet-recent-block-streams`
- Previewnet: `hedera-previewnet-recent-block-streams`

> **Note:** Cloud bucket publishing does not begin during Stage 1. Buckets are populated starting
> at Stage 2.

#### 5.2 Path Structure Implementation

The path structure for block streams will be:
{network}-{yyyyMMddHHmm}/block/0000/0000/0000/0000/001.blk.zstd

Block files will use a 19-digit zero-padded block number format. Files will use Zstandard
compression and the `.blk.zstd` extension. For mainnet, the network identifier will be a static
`mainnet` with no `-` suffix. For other networks, the network identifier will take the form of
`{network}-{yyyyMMddHHmm}`. The `network` will be the network name and the `yyyyMMddHHmm` will be
the timestamp of the last network reset. Here's an example of a resettable network:

testnet-202402010901/block/0000/0000/0000/0000/000.blk.zstd
testnet-202402010901/block/0000/0000/0000/0000/001.blk.zstd
...
testnet-202402010901/block/0000/0000/0000/9999/999.blk.zstd
testnet-202402010901/block/0000/0000/0001/0000/000.blk.zstd
testnet-202601011302/block/0000/0000/0000/0000/000.blk.zstd
testnet-202601011302/block/0000/0000/0000/0000/001.blk.zstd

In this example, the testnet data started at block zero on 2024-02-01 and continued until block
1,000,000 when it was reset on 2026-01-01. Mirror nodes automatically detect the reset by
periodically listing and sorting the network identifiers in the top-level folder and choosing the
most recent one.

### 6\. Mirror Node

#### 6.1 Stage 1 Configuration and Fallback

During Stage 1, mirror nodes may be optionally configured to receive WRBs. The following behavior
applies:

- Mirror nodes are **not required** to receive WRBs during Stage 1. Operators may continue
  receiving record streams exclusively with no action required.
- Mirror nodes **configured** to receive WRBs will connect to the designated Stage 1 block nodes
  and ingest WRBs in parallel with or in lieu of record stream processing, depending on operator
  configuration.
- If a mirror node configured to receive WRBs cannot connect to a block node, fails to receive a
  WRB within a configurable timeout, or receives a WRB that fails validation, it will
  **automatically fall back to record stream processing**. This fallback is silent and requires no
  operator intervention.
- The fallback logic ensures that **no data loss can result from participating in Stage 1 WRB
  testing**.

#### 6.2 Block Ingestion (Stage 2)

At Stage 2, mirror nodes will implement logic to:

- Support the new block path structure.
- Support cutover for both block stream files in the bucket and streaming the data directly from
  block nodes.
- Periodically check for the existence of block stream data when no record stream data is available
  for a period of time.
- Switch from record stream processing to block stream processing permanently.
- Support retry mechanisms for failed block ingestion.
- Store block data in a compatible format with existing database schemas.

#### 6.3 Block Verification

Mirror nodes will:

- Implement block proof verification using Threshold Signature Scheme (TSS).
- Store and use the ledger ID for block verification.
- Implement retry mechanisms to handle cases where block verification fails.

> **Note:** During Stage 1, block proofs are RSA-derived (not TSS). 

#### 6.4 Bootstrap and Recovery

Mirror nodes will:

- Implement logic to process both block and record files.
- Implement logic to update the timestamp-based startup logic to use blocks rather than timestamps.
- Take PostgreSQL snapshots before the Stage 2 transition to enable mirror node bootstrapping.
- Support recovery in cases where block processing fails.

### 7\. Test Plan and Verification

#### 7.1 Stage 1 Verification

1. Verification that consensus nodes produce WRBs correctly alongside uninterrupted record streams.
2. Verification that designated block nodes receive and store WRBs without errors.
3. Verification that mirror nodes configured to receive WRBs ingest and validate them correctly.
4. Verification that mirror nodes automatically fall back to record streams upon WRB receipt
   failure or validation failure.
5. Verification that record stream production is unaffected by any WRB streaming errors.
6. Evaluation against all Stage 1 readiness gate criteria prior to authorizing Stage 2.

#### 7.2 Stage 2 Verification

1. Verification of logic for writing block streams.
2. Verification of block production and correctness.
3. Verification of ingestion and cloud storage.
4. Verification of block proof creation and verification following node upgrade.
5. Verification of Mirror Node transition detection and cutover capability.

### 8\. Disaster Recovery Considerations

#### 8.1 Stage 1 Risks

- **WRB streaming failure**: Consensus nodes must continue producing record files regardless of
  WRB streaming status. No operator action is required; record streams serve as the fallback.
- **Mirror node WRB validation failure**: Mirror nodes fall back to record streams automatically.
  Operators should monitor fallback events and report them to aid in defect diagnosis.
- **Misconfigured mirror node failing to fall back**: Mirror node configuration must be validated
  before Stage 1 to ensure fallback logic is active and correctly configured.

#### 8.2 Pre-First-Block Risks (Stage 2)

- Node failure during cutover.
- Uploader misconfiguration (incorrect path, cloud provider, etc.).
- No blocks being produced.
- Record or event streams not stopping as intended.

#### 8.3 First Block Production Risks (Stage 2)

- Incorrect running hash.
- Incorrect block number.
- Malformed block data or schema.
- Invalid block proof (WRAPS or TSS).
- Node reconnection issues after a maintenance window.

## Backwards Compatibility

This transition introduces a fundamental change in how transaction data is exposed by a Hiero
network, but it does so in two stages to preserve compatibility throughout.

**During Stage 1**, no cryptographic or operational compatibility is broken. Record streams
continue to be produced and remain the authoritative source of network data. Mirror nodes not
configured for WRB testing are entirely unaffected. Mirror nodes configured for WRB testing retain
full fallback safety to record streams.

**At Stage 2**, the hard cryptographic transition occurs. To maintain backward compatibility:

- Historical record stream data will remain available in its current format and location for the
  foreseeable future.
- Mirror nodes will support both record stream and block stream processing.
- The first block will maintain cryptographic continuity with the last record. This includes that
  the "previous hash" for the first block after cutover will be reported as the record running hash
  of the last record file.
- There will be no impact on the mirror node API structure or content.

Long-term, as the ecosystem transitions to block nodes, historical record streams and cloud bucket
access may eventually be deprecated, but this would require a separate HIP and migration plan.

## Security Implications

The transition to block streams introduces several security considerations:

1. **Data Integrity**: Ensuring the root hash continuity between record streams and block
   streams is critical to maintain the cryptographic integrity of the ledger.

2. **Block Verification**: Mirror nodes must properly implement the TSS verifier to validate block
   proofs at Stage 2. During Stage 1, RSA-derived proof verification is used and must also be
   correctly implemented.

3. **Stage 1 Authority Boundary**: During Stage 1, WRBs contain identical content and share the same
   cryptographic authority as record files. Mirror nodes and consumers _may_ treat Stage 1 WRBs
   as a substitute for record stream finality, but are not required to do so.

4. **Disaster Recovery**: Proper procedures must be in place to handle node failures or other
   issues during both stages of the cutover.

5. **Data Availability**: Ensuring no transactions are lost during the transition is critical for
   maintaining network trust. The two-stage approach, with automatic fallback in Stage 1, directly
   addresses this risk.

## How to Teach This

The transition from record streams to block streams represents a significant architectural change
for the Hiero network. To help the community understand this change:

1. **For Mirror Node Operators:**

    - Explain the two-stage cutover and what action, if any, is required at each stage.
    - Explain how to optionally configure a mirror node to receive WRBs during Stage 1 and what
      the automatic fallback behavior means in practice.
    - Provide detailed documentation on the new block file format and paths.
    - Explain how to verify access to the new block stream buckets.
    - Explain how to configure connectivity to block nodes, if applicable.
    - Offer guidance on handling the transition period.

2. **For Developers:**

    - Explain the conceptual shift from records to blocks.
    - Clarify that Stage 1 does not change API behavior or data availability for downstream
      consumers.
    - Emphasize that the Mirror Node behavior and APIs remains unchanged
    - Emphasize that the Consensus Node HAPI interface is unchanged.
    - Document how transaction data will be organized in the new format.
    - Provide examples of how to process block data.

3. **For Node Operators:**

    - Detail the changes required to consensus node configuration for each stage.
    - Explain the new uploader paths and configurations.
    - Clarify that record file production continues uninterrupted through Stage 1.
    - Provide a timeline and checklist for both Stage 1 and Stage 2 upgrades.

## Rejected Ideas

The cutover approach has morphed over time. Initially, it was thought block nodes would be ready
much later than block streams, so it might be best to cutover to block streams stored in cloud
storage first. Since block streams have taken some time to be finalized, it was decided to directly
cutover to block nodes and skip cloud storage. This approach changed to a hybrid approach where
block streams would be available from both block nodes and cloud storage. The current approach
further refines this by introducing a pre-cutover Stage 1 (WRB Cutover) to allow infrastructure
validation and integration testing before committing to the irreversible TSS cryptographic cutover.
