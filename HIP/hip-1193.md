## Abstract

This HIP defines the requirements and implementation details for transitioning the Hedera network from record and event
streams to block streams. This transition is a critical step toward enhancing blockchain compatibility, improving data
integrity, and enabling future network optimizations such as state proofs. The proposal outlines the coordinated changes
required across consensus nodes, mirror nodes, and supporting infrastructure to ensure a clean cutover without service
disruption. It specifies new buckets, storage paths, file formats, and mechanisms to handle the transition.

## Motivation

As highlighted in the Block Streams [HIP 1056](https://hips.hedera.com/hip/hip-1056), the Hedera network currently
produces record streams as its primary method of exposing transaction data to mirror nodes and external consumers. While
functional, this approach diverges from standard blockchain architectures that organize data into blocks. By
transitioning to block streams, Hedera will:

1. Improve blockchain compatibility and interoperability with existing blockchain tools and infrastructure.
2. Provide better data integrity through block-level proofs and signatures.
3. Reduce storage costs through optimized block organization and eventual deduplication.
4. Enable future improvements like block-based state proofs and block node architecture.
5. Simplify the developer experience by aligning with familiar blockchain concepts.

This transition is a prerequisite for the planned [block node](https://hips.hedera.com/hip/hip-1081) architecture and
represents an important step in Hedera's technical evolution.

## Rationale

The transition from record streams to block streams requires careful planning to ensure network continuity, data
integrity, and backward compatibility. Block stream data will be available both from block nodes and limited history
cloud buckets. This HIP proposes a clean cutover approach with the following key design decisions:

## Decouple Block Stream and Block Nodes

Dual publishing the block streams to both cloud storage and block nodes decouples the rollout of the two. There is
some risk there might not be enough block nodes available at cutover to support the number of mirror nodes consuming
stream data. By publishing the most recent data to a new bucket in cloud storage, it allows existing mirror nodes to
continue without modification. The cloud storage can scale to an infinite number of consumers as it has done in the past
for record streams. When block nodes reach wider availability, operators can choose to switch over to enjoy the
benefits they bring like lower latencies.

### Cloud Storage

New cloud storage buckets will be created to hold block stream data. These buckets will be limited-history and only
contain the most recent data. Syncing data from genesis will require connecting to block nodes or importing a state
snapshot. The decision to create new buckets instead of reusing the existing buckets was made based on the need to use
retention policies to efficiently prune historical data.

The mirror node will be updated to automatically use the new buckets when configured to use one of the three public
networks. Operators only need to ensure their existing cloud storage credentials have the necessary permissions to read
the new buckets before the cutover.

## Resettable and Ephemeral Networks

The new block stream files will follow a revised path structure that includes a network identifier. The network
identifier will allow for additional flexibility in non-mainnet environments including ephemeral networks, multi-network
buckets, and automatic resets. Having the network name included in the path allows for the option of having data from
multiple networks to be stored in the same bucket. This flexibility will reduce the maintenance burden on operations
teams.

## Efficient Block Retrieval

When connected directly to a block node, blocks will be directly pushed to consumers like mirror nodes via a streaming
gRPC API as defined in HIP-1081. The block streams uploaded to cloud storage will be compressed and named for efficient
retrieval. Block files will use the `.blk.zstd` extension and Zstandard compression to provide maximum data compression.

The file name will be the block number instead of a date like in the record stream. This will avoid costly list
operations and allow mirror nodes to directly download the next sequential block number. The 19-digit block number will
be partitioned into 4-digit paths to reduce the amount of data that resides in a single directory in cloud storage. This
will allow for efficient listing of folders for use in debugging or pruning.

### Bucket Structure and File Format

The new block stream files will follow a revised path structure that includes a network identifier. This identifier will
allow for additional flexibility in non-mainnet environments including ephemeral networks, multi-network buckets, and
automatic resets. Having the network name included in the path allows for the option of having data from multiple
networks to be stored in the same bucket in pre-production environments.

## Seamless Mirror Node Transition

The mirror node will automatically detect a cutover occurred by periodically checking for block stream data if no record
streams have been received within a period of time. If no block node connection information is available either via
manual configuration or HIP-1137 Registered Node transactions, the mirror node will fall back to bucket-based retrieval.
If it detects block stream data in either a configured block node or cloud storage, it will permanently transition to
using block streams. This approach:

- Allows mirror nodes to detect the transition without configuration changes.
- Prevents data loss during the transition.
- Eases mirror node operations, enabling mirror nodes to update the mirror node version before the block stream cutover
  and have seamless service continuity.

## User stories

1. As a mirror node operator, I want a means to automatically transition from record stream processing to block stream
   processing so that I don't miss any transactions during the cutover or cause downtime in my application.

2. As a mirror node operator, I want to understand the new block file format and paths so that I can adjust my
   infrastructure to process block streams efficiently.

3. As a developer building on Hedera, I want to understand how the transition affects data availability and processing
   so that my applications continue to function correctly.

4. As a consensus node operator, I want to know exactly what changes are needed to support block streams so that I can
   prepare for the upgrade.

5. As a mirror node operator, I need to know what version of the mirror node software I need to be running and have
   enough time to upgrade to that version before the cutover.

6. As a network user, I want the transition to be seamless with no impact on transaction processing or data
   availability.

7. As a consumer of mirror node data and associated metrics, I want the transition to be seamless with no impact on how
   I consume mirror node data.

## Specification

### 1\. Ledger ID Generation

To prepare for the transaction, some work will take place in the release prior to the cutover:

- Consensus nodes will perform the TSS ceremony.
- The ledger ID will be generated and externalized by the consensus nodes via a synthetic
  `LedgerIdPublicationTransaction` as defined in HIP-1200.
- Block nodes and mirror nodes will receive this transaction and persist the ledger ID in their respective databases.
- Mirror node code will be updated to hardcode this ledger ID per network for any operators that start after this
  ledger ID publication.

### 2\. Cutover Point Definition

The transition from record streams to block streams will occur during a Hedera mainnet update boundary, with the
following requirements:

- Occurs after a Hedera Freeze Upgrade transaction during the Hedera mainnet update window.
- The current round comes to consensus.
- The final record stream will be completed.
- Block continuity will be maintained by carrying forward:
  - The correct block number (incremented by one from the last record).
  - The last running hash value will be the first item in the next block.

### 3\. Consensus Node

#### 3.1 Record and Event Stream Termination

Consensus nodes will:

- Perform a Freeze operation and stop accepting transactions.
- Freeze round comes to consensus.
- Ensure all signatures are collected and files are properly finalized.
- Produce the final record file.
- Stop producing both record and event streams at the designated cutover point.
- Shutdown uploaders as uploading responsibility is moved to block nodes.

#### 3.2 Block Stream Initialization

Consensus nodes will:

- Construct the first block.
- Apply the correct running hash and block number from the final record.
- Sign the block using TSS and the ledger ID previously generated.
- Push block streams to the configured block nodes.

### 4\. Block Node

- Upload block streams to the new buckets following the new path structure.
- Deduplicate block streams by block number so only one copy is stored in the bucket.
- Immediate streaming access to exported consensus network data.

### 4\. Cloud Storage

#### 5.1 Bucket Creation

The following are the three new buckets:

- Mainnet: `hedera-mainnet-recent-block-streams`
- Testnet: `hedera-testnet-recent-block-streams`
- Previewnet: `hedera-previewnet-recent-block-streams`

#### 5.2 Path Structure Implementation

The path structure for block streams will be:

```
{network}/0000/0000/0000/0000/001.blk.zstd
```

For mainnet, the network identifier will be a static `mainnet`. For other networks, the network identifier will take the
form of `{network}-{yyyyMMddHHmm}`. The `network` will be the network name and the `yyyyMMddHHmm` will be the timestamp
of the last network reset. Here's an example of a resettable network:

```
testnet-20260803/0000/0000/0000/0000/001.blk.zstd
```

- The network value will identify the network (mainnet, testnet, etc.)
- Block files will use a 19-digit zero-padded block number format.
- Files will use Zstandard compression and the `.blk.zstd` extension

### 6\. Mirror Node

#### 6.1 Processing Logic

Mirror nodes will implement logic to:

- Implement logic to periodically check for the existence of block stream data when no record stream data is available
  for a period of time.
- Switch from record stream processing to block stream processing
- Support the new block path structure.
- Support retry mechanisms for failed block ingestion
- Implement block proof verification using Threshold Signature Scheme (TSS).
- Store block data in a compatible format with existing database schemas

#### 6.2 Block Verification

Mirror nodes will:

- Implement block proof verification using Threshold Signature Scheme (TSS)
- Implement retry mechanisms to handle cases where block verification fails. Should retry the same block number from
  another node until correctly processed.

#### 6.3 Bootstrap and Recovery

Mirror nodes will:

- Implement logic to process both block and records files.
- Implement logic to update the timestamp-based startup logic to use blocks rather than timestamps.
- Take PostgreSQL snapshots before the transition to enable mirror node bootstrapping.
- Support recovery in cases where block processing fails.

### 7\. Test Plan and Verification

The end-to-end test plan will include:

1. Verification of logic for writing block streams.
2. Verification of block production and correctness.
3. Verification of ingestion and cloud storage.
4. Verification of block proof creation and verification following node upgrade.
5. Verification of Mirror Node transition detection and cutover capability.

### 8\. Disaster Recovery Considerations

The following risk scenarios must be addressed:

#### 8.1 Pre-First-Block Risks

- Node failure during cutover.
- Uploader misconfiguration (incorrect path, cloud provider, etc.).
- No blocks being produced.
- Record or event streams not stopping as intended.

#### 8.2 First Block Production Risks

- Incorrect running hash.
- Incorrect block number.
- Malformed block data or schema.
- Invalid block proof.
- Node reconnection issues after a maintenance window.

## Backwards Compatibility

This transition introduces a fundamental change in how transaction data is exposed by the Hedera network.

To maintain backward compatibility:

* Historical record stream data will remain available in its current format and location for the foreseeable future.
* Mirror nodes will support both record stream and block stream processing.
* The first block will maintain cryptographic continuity with the last record.
* There will be no impact on mirror node API structure or content.

Long-term, as the ecosystem transitions to block nodes, historical record streams and cloud bucket access may eventually
be deprecated, but this would require a separate HIP and migration plan.

## Security Implications

The transition to block streams introduces several security considerations:

1. **Data Integrity**: Ensuring the running hash continuity between record streams and block streams is critical to
   maintain the cryptographic integrity of the ledger.

2. **Block Verification**: Mirror nodes must properly implement the TSS verifier to validate block proofs.

3. **Disaster Recovery**: Proper procedures must be in place to handle node failures or other issues during the cutover.

4. **Data Availability**: Ensuring no transactions are lost during the transition is critical for maintaining network
   trust.

## How to Teach This

The transition from record streams to block streams represents a significant architectural change for the Hedera
network. To help the community understand this change:

1. **For Mirror Node Operators:**

   - Provide detailed documentation on the new block file format and paths.
   - Explain how to verify access to the new block stream buckets.
   - Explain how to configure connectivity to block nodes, if applicable.
   - Offer guidance on handling the transition period.

2. **For Developers:**

   - Explain the conceptual shift from records to blocks.
   - Document how transaction data will be organized in the new format.
   - Provide examples of how to process block data.

3. **For Node Operators:**

   - Detail the changes required to consensus node configuration.
   - Explain the new uploader paths and configurations.
   - Provide a timeline and checklist for the upgrade.

## Rejected Ideas

The cutover approach has morphed over time. Initially, it was thought block nodes would be ready much later than block
streams, so it might be best to cutover to block streams stored in cloud storage first. Since block streams have taken
some time to be finalized, it was decided to directly cutover to block nodes and skip cloud storage. This approach
changed to the current hybrid approach where the block streams will be available by both block nodes and cloud storage.
