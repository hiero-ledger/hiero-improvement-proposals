## Abstract

This HIP proposes a Post-Quantum Cryptography (PQC) readiness assessment framework for the Hiero ecosystem. As quantum computing advances toward cryptographically-relevant capability (estimated 2028–2033), all ED25519 and ECDSA-secp256k1 keys currently used across Hiero accounts, topics, tokens, and smart contracts will become vulnerable to Shor's algorithm. This proposal defines: (1) a classification rubric for cryptographic algorithm vulnerability, (2) a readiness scoring methodology, (3) tooling specifications for auditing on-chain key exposure, and (4) a migration path toward NIST FIPS 203/204/205 compliant algorithms (ML-KEM, ML-DSA, SLH-DSA).

## Motivation

### The Quantum Threat Timeline

NIST finalized the first three post-quantum cryptography standards in August 2024 (FIPS 203, 204, 205). The NSA's CNSA 2.0 suite mandates PQC adoption for national security systems by 2027. The SWIFT network requires PQC readiness by 2027. The EU Cyber Resilience Act and the EU AI Act (effective August 2026) include provisions for cryptographic agility.

### Hiero's Current Exposure

Every key in the Hiero ecosystem today is quantum-vulnerable:

| Algorithm | Usage in Hiero | Quantum Vulnerability |
|-----------|---------------|----------------------|
| ED25519 | Default account keys, topic keys, token keys | CRITICAL — broken by Shor's algorithm |
| ECDSA-secp256k1 | EVM-compatible accounts, smart contracts | CRITICAL — broken by Shor's algorithm |
| RSA (HIP-1334) | Private message box encryption | CRITICAL — broken by Shor's algorithm |
| ECIES (HIP-1334) | Private message box encryption | CRITICAL — broken by Shor's algorithm |

No Hiero SDK, CLI, or network node currently supports post-quantum key types. No HIP has previously addressed this gap.

### Why Now

The "harvest now, decrypt later" attack vector means that adversaries can record encrypted traffic today and decrypt it once quantum computers are available. For financial infrastructure — Hiero's primary use case — this means sensitive transaction data recorded on-chain today may be retroactively compromised. Assessment tooling must exist before migration tooling, because organizations need to understand their exposure before they can plan remediation.

## Rationale

This HIP takes an assessment-first approach rather than proposing new key types directly (which would require changes to the Hedera consensus node, SDK key management, and mirror node indexing). Assessment tooling can be built entirely at the application layer using existing SDK capabilities, providing immediate value while the community designs the deeper protocol changes needed for PQC key support.

The readiness scoring methodology is adapted from NIST's Cryptographic Agility guidance (SP 800-131A Rev 3) and the CBOM (Cryptography Bill of Materials) specification used by IBM's Hyperion/Theia tooling.

## User Stories

1. **As a Hiero application developer**, I want to audit my application's accounts to understand which keys are quantum-vulnerable, so I can plan migration timelines aligned with CNSA 2.0 deadlines.

2. **As an enterprise compliance officer**, I want a machine-readable report of my organization's quantum exposure across all Hiero accounts and topics, so I can include it in our risk assessment for regulators.

3. **As a Hiero CLI user**, I want to run `hcli pqc audit` and see which of my managed accounts use quantum-vulnerable key types, along with a readiness score and recommended actions.

4. **As a Hiero SDK developer**, I want a reference classification of algorithm vulnerability levels, so I can implement PQC-aware key management in my applications.

## Specification

### 1. Cryptographic Algorithm Classification

All algorithms are classified into five vulnerability tiers:

| Tier | Label | Description | Examples |
|------|-------|-------------|----------|
| 0 | `PQC_READY` | Post-quantum algorithm, NIST standardized | ML-DSA-65, ML-KEM-768, SLH-DSA |
| 1 | `STRONG` | Classical algorithm, no known quantum advantage beyond Grover | AES-256, SHA-3-256 |
| 2 | `MODERATE` | Classical algorithm, partial quantum speedup | AES-128 (Grover halves effective key length) |
| 3 | `WEAK` | Algorithm with known classical weaknesses OR reduced quantum resistance | RSA-2048, 3DES |
| 4 | `CRITICAL` | Fully broken by Shor's algorithm on a CRQC | ED25519, ECDSA-secp256k1, RSA, ECIES, DH |

### 2. Quantum Readiness Score (QRS)

A composite score from 0–100 measuring an entity's quantum preparedness:

```
QRS = (key_score × 0.40) + (algorithm_diversity × 0.20) + (rotation_readiness × 0.20) + (compliance_alignment × 0.20)
```

Where:
- **key_score** (0–100): Weighted average of all key vulnerability tiers (Tier 0 = 100, Tier 4 = 0)
- **algorithm_diversity** (0–100): Penalizes single-algorithm dependency; rewards hybrid approaches
- **rotation_readiness** (0–100): Measures whether admin keys exist for rotation (accounts without admin keys cannot migrate)
- **compliance_alignment** (0–100): Alignment with CNSA 2.0 timeline (days until 2027 deadline)

### 3. Audit Output Schema

Machine-readable JSON output for each audited entity:

```json
{
  "entityId": "0.0.123456",
  "entityType": "account",
  "network": "mainnet",
  "auditTimestamp": "2026-03-24T00:00:00Z",
  "keys": [
    {
      "keyType": "admin",
      "algorithm": "ED25519",
      "vulnerabilityTier": 4,
      "vulnerabilityLabel": "CRITICAL",
      "canRotate": true
    }
  ],
  "quantumReadinessScore": 12,
  "recommendations": [
    "Admin key uses ED25519 (Tier 4: CRITICAL). Plan migration to ML-DSA-65 when supported.",
    "Account has admin key — key rotation is possible when PQC support is added."
  ],
  "complianceFlags": {
    "cnsa2_2027": false,
    "nist_fips_203": false,
    "nist_fips_204": false
  }
}
```

### 4. CLI Plugin Specification

A `pqc` plugin for hiero-cli with three commands:

| Command | Description |
|---------|-------------|
| `hcli pqc audit` | Audit managed accounts/topics for quantum vulnerability |
| `hcli pqc score` | Calculate Quantum Readiness Score for a specific entity |
| `hcli pqc report` | Generate a full compliance report (JSON, CSV, human-readable) |

### Impact on Mirror Node

No mirror node changes required. The audit tooling reads key types from existing account/topic info queries via the SDK.

### Impact on SDK

No SDK changes required for the assessment tooling. Future HIPs proposing PQC key type support will require SDK changes.

## Backwards Compatibility

This HIP is fully backwards-compatible. It introduces read-only assessment tooling that does not modify any on-chain state or change any existing APIs.

## Security Implications

1. The audit tooling reveals which accounts use which key types. Reports should be treated as sensitive security information.
2. The Quantum Readiness Score is an assessment tool, not a security guarantee. A high score does not mean an entity is quantum-safe — it means the entity is better prepared for migration.
3. This HIP does not introduce any new key types or cryptographic operations.

## How to Teach This

1. Add a "Quantum Readiness" section to the Hiero CLI documentation explaining the `pqc` plugin commands.
2. Publish a blog post explaining the quantum threat to blockchain infrastructure and how Hiero is preparing.
3. Include QRS scores in the Hiero dashboard for enterprise customers.

## Reference Implementation

A reference implementation is provided as a hiero-cli plugin: [PR #1691](https://github.com/hiero-ledger/hiero-cli/pull/1691)

The plugin implements all three commands (`audit`, `score`, `report`) following the standard hiero-cli plugin architecture, with Zod input/output schemas, Handlebars templates, and unit tests.

## Rejected Ideas

1. **Proposing new PQC key types in this HIP**: Too large a scope. Key type changes require consensus node modifications, SDK updates, and mirror node indexing changes. Assessment tooling should exist first.
2. **Using external certificate scanning**: SSL certificate scanning of external domains is unrelated to Hiero's on-chain cryptographic posture. On-chain key auditing is the correct scope.
3. **Mandatory migration deadlines**: Premature without PQC key type support in the protocol. Assessment comes first.

## Open Issues

1. Should the QRS scoring weights be configurable per-organization?
2. Should the plugin support threshold key analysis (e.g., a 2-of-3 multisig where one key is PQC)?
3. How should the scoring handle KeyList and ThresholdKey structures?
4. Should a future HIP define a PQC key type for Hiero accounts, building on this assessment framework?
