# HIP-1496 - Globally Synchronized Network Upgrade Process

## Abstract

This proposal introduces a new deployment model and network upgrade process for Hedera consensus nodes (CNs). The goal
is to enable globally distributed consensus nodes to freeze the network synchronously at a predetermined time, upgrade
to a desired software version, and restart autonomously — with minimal risk of network disruption and delay.

In short: the new process shrinks the upgrade package by 99%, gives every council member cryptographic confidence in
what software is running, makes infrastructure upgrades part of the same coordinated freeze window, and replaces fragile
bash scripts with observable, resumable Kubernetes-native automation.

## Motivation

### Background

To motivate the need for the new approach, we first describe the current deployment model and upgrade protocol,
highlighting its limitations.

**Why Freeze the Network During an Upgrade?**

A network freeze is required to ensure that all nodes are running the exact same software version before resuming
consensus operations. If nodes are running different versions, they may process transactions differently, leading to
state hash mismatches. Such mismatches are catastrophic: the network would halt, as consensus cannot be reached when
nodes disagree on the state. By freezing the network, performing the upgrade, and then restarting all nodes together,
we guarantee state consistency and prevent consensus failures.

**Docker Compose Based Deployment Model**

Currently, each Hedera Consensus Node (CN) runs on a bare metal machine inside a Docker container, alongside stream file
uploader service containers, all orchestrated via Docker Compose. A native daemon service, the **Upgrade Controller
(UC)**, runs on the host machine alongside these containers. The UC monitors marker files written by the CN during the
upgrade protocol and orchestrates the container lifecycle, namely — rebuilding images, executing lifecycle scripts, and
restarting containers at the appropriate time.

**Network Upgrade Package**

Before an upgrade window, an upgrade package (`build-<VERSION>.zip`, e.g. `build-v0.68.6.zip`, ~100 MB) is prepared,
signed by council members, and uploaded to the Hedera File System (HFS, FileID `0.0.150`) using HAPI file update/append
transactions. Due to per-transaction payload limits, the package is split into many smaller chunks, requiring 3200
transactions (assuming 32Kb payload size per transaction) to upload, taking ~10 minutes using an optimized file upload
protocol implemented in a tool
called [yahcli](https://github.com/hiero-ledger/hiero-consensus-node/tree/main/hedera-node/yahcli). This process is
time-consuming, costly, and may need multiple attempts to complete successfully.

***The size of this upgrade package is one of the key motivations for the new deployment model proposed later in this
document, which eliminates the need to upload JAR files to the network.***

The upgrade package generally includes:

- `data/apps/HederaNode.jar` — new version of the consensus node software
- `data/config/` — various configuration files
- `data/lib/*.jar` — dependency JAR files
- `immediate.sh` — executed during the PREPARE_UPGRADE phase for pre-freeze preparation
- `during_freeze.sh` — executed after the CN enters the frozen state, before the new container is started
- `settings.txt` — properties controlling consensus node behaviour
- `log4j2.xml` — logging configuration
- `VERSION` — version and commit hash of the software

```markdown
build-v0.68.6.zip
├── data
│ ├── apps
│ ├── config
│ └── lib
├── during_freeze.sh
├── genesis_network.json
├── immediate.sh
├── log4j2.xml
├── settings.txt
└── VERSION
```

**Network Upgrade Protocol**

Once the upgrade package is uploaded, the upgrade protocol is initiated through a sequence of freeze transactions signed
by the council members to freeze and then upgrade the consensus nodes.

The protocol involves three transaction types (`PREPARE_UPGRADE`, `FREEZE_UPGRADE`, `FREEZE_ABORT`) that drive the CN
through certain state transitions as described below.

The key point is that the CN remains in `ACTIVE` platform status during the **Upgrade Prepared** and
**Freeze Scheduled** phases, and only transitions to `FREEZING` when the scheduled
freeze time is reached. The `FREEZE_ABORT` transaction can return the node to `ACTIVE` status from either of the first
two phases, but once the node enters `FREEZING`, the process is irreversible. The overall state transition flow is:

`ACTIVE → Upgrade Prepared → Freeze Scheduled → FREEZING → FREEZE_COMPLETE → ACTIVE (Upgraded)`

**Platform Status State Machine**

Internally CN goes through its own state transitions as below during network freeze:

```markdown

CATASTROPHIC_FAILURE(11) ← unrecoverable error

# state transition during freeze

STARTING_UP (1)
│
▼
REPLAYING_EVENTS (7)
│
▼
OBSERVING (8)
│
▼

│
▼

│
freeze time reached
│
▼
FREEZING (5)
│

│
▼
FREEZE_COMPLETE (6)
```

Key points:

- ACTIVE — normal operation, gossiping, creating events, accepting transactions
- FREEZING — freeze time reached, processing the freeze round, still gossiping to distribute signatures, but no new
  transactions
- FREEZE_COMPLETE — freeze state written to disk, now_frozen.mf written (if upgrade pending), UC takes over
- BEHIND — node is out of sync, needs reconnect (not gossiping)
- CHECKING — rejoining the network after startup/reconnect, gossiping and creating events but not yet accepting app
  transactions
- **Upgrade Prepared and** **Freeze Scheduled** are not distinct platform statuses — the platform remains in `ACTIVE`
  status during both of those phases. The platform status only changes when the freeze round arrives (
  `ACTIVE → FREEZING`) and then when the freeze state is written to disk (`FREEZING → FREEZE_COMPLETE`). The "Upgrade
  Prepared" and "Freeze Scheduled" are logical phases of the upgrade protocol tracked through persistent state (
  `updateFileHash` and `freezeTime`) and marker files, not platform status transitions.
- A `FREEZE_ABORT` transaction can return the node from either **Upgrade Prepared** or **Freeze Scheduled** back to *
  *ACTIVE**. Once the network enters the **FREEZING** state, the process is irreversible.

Below we describe what happens for various “freeze” type transactions (
see [services/freeze.proto](https://github.com/hashgraph/hedera-protobufs/blob/8c27786cec93abab974309074feaef9b48a695b7/services/freeze.proto)):

**PREPARE_UPGRADE Tx**

A `PREPARE_UPGRADE` transaction signals the intent of a near-future network upgrade, typically sent ~2 days ahead of the
desired upgrade window. The transaction body contains the `freeze_type` set to `PREPARE_UPGRADE`, the upgrade package
file ID (e.g., `0.0.150`), and its SHA-384 hash. The `start_time` field is not used for this transaction type.

Upon handling this transaction, each consensus node:

1. Validates that the referenced upgrade file exists and its SHA-384 hash matches the hash in the transaction.
2. Stores the file hash in persistent state, marking the upgrade as "prepared."
3. Extracts the upgrade archive asynchronously into the upgrade artifacts directory (`data/upgrade/current/`). If DAB (
   Dynamic Address Book) is enabled, it also generates `config.txt` and PEM certificate files for all active nodes.
4. Writes the `execute_immediate.mf` marker file upon successful extraction. If extraction fails, no marker is written.

The Upgrade Controller (UC) detects the `execute_immediate.mf` marker and performs its preparation steps — validating
file integrity, executing `immediate.sh`, and rebuilding docker images — without tearing down the running container.

If a node reconnects after this transaction has been processed by the network, it catches up by re-reading the upgrade
file, validating the hash, and re-running the extraction.

**FREEZE_UPGRADE Tx**

A `FREEZE_UPGRADE` transaction **schedules** the network freeze at a specified future time. The transaction body
contains the `freeze_type` set to `FREEZE_UPGRADE` and a `start_time` that must be strictly after the current consensus
time.

Upon handling this transaction, each consensus node:

1. Stores the scheduled freeze time in persistent state.
2. Writes the `freeze_scheduled.mf` marker file containing the freeze timestamp (epoch seconds).

The node does **not** freeze immediately — it continues normal transaction processing until the scheduled time arrives.

The UC detects the `freeze_scheduled.mf` marker, parses the freeze timestamp, and revalidates the upgrade package.

If a node reconnects after this transaction has been processed, it catches up by re-writing the `freeze_scheduled.mf`
marker from the freeze time stored in the platform state.

***NOW_FROZEN State***

The actual freeze is enforced at the platform layer. Each consensus round, the platform checks whether the round's
consensus timestamp has reached or passed the scheduled freeze time. When it has:

1. The freeze round is **fully processed** — all transactions in that round are handled normally.
2. The platform status transitions to `FREEZING`, and `lastFrozenTime` is set to `freezeTime` (preventing re-freeze on
   restart).
3. All subsequent consensus rounds are **dropped** — the node stops processing new transactions.
4. Once the freeze round state is persisted to disk, the node writes the `now_frozen.mf` marker file, but only if a
   software upgrade is pending (i.e., not for `FREEZE_ONLY`).
5. The platform status transitions to `FREEZE_COMPLETE`.

The UC detects the `now_frozen.mf` marker and proceeds with its upgrade steps — backing up files, tearing down the
container, executing `during_freeze.sh`, and restarting with the new software version.

**FREEZE_ABORT Tx**

A `FREEZE_ABORT` transaction cancels a pending upgrade. It can be sent any time before the scheduled freeze time is
reached.

Upon handling this transaction, each consensus node:

1. Clears the stored freeze time and update file hash, fully resetting the upgrade state.
2. Writes the `freeze_aborted.mf` marker file.

The UC detects the `freeze_aborted.mf` marker and performs cleanup — removing scheduled tasks, terminating any running
`immediate.sh` processes, and cleaning the upgrade artifacts directory.

Key points about FREEZE_ABORT:

- **FREEZE_ABORT** can be issued from either **Upgrade Prepared** or **Freeze Scheduled** states, returning the node to*
  *ACTIVE**.
- **FREEZE_ABORT** is **not possible** once the consensus time has reached the scheduled freeze time (i.e., once the
  node enters `FREEZING`).

### Why Change Is Needed

The current Docker Compose based deployment model and upgrade process has several significant limitations that this
proposal addresses:

### 1. Large Upgrade Package Uploaded via HFS

The current upgrade package (`build-<VERSION>.zip`) is approximately 100 MB, consisting primarily of JAR files (
`HederaNode.jar` and dependency libraries). This package must be uploaded to the Hedera File System (HFS, FileID
`0.0.150`) using HAPI file update/append transactions. Due to per-transaction payload limits, the package is split into
many small chunks, requiring ~3200 transactions to upload a single upgrade package.

This creates several problems:

- **Cost**: Each transaction incurs fees, making the upload process expensive.
- **Time**: The upload process is slow due to the sheer number of transactions required.
- **Reliability**: Network issues or transaction failures during the upload may require restarting the entire process,
  compounding both cost and time.
- **Operational burden**: Council members must coordinate to sign and submit thousands of transactions, with manual
  monitoring for failures and retries.

By replacing the large JAR-based zip with a small manifest (~1 KB) that references pre-built container images in
external registries, the number of upload transactions drops to single digits, eliminating the cost, time, and
reliability concerns.

### 2. No Support for Deterministic Container Images

In the current model, Docker images are built locally on each host machine during every upgrade. The Node Upgrade
Daemon (UC) executes a multi-step image build process on the host machine during upgrade — constructing base images,
copying JAR files, and assembling the final consensus node image. This approach has several drawbacks:

- **Non-reproducibility**: Images built on different hosts may differ due to environment variations, making it
  impossible to guarantee that all nodes run identical container image.
- **Build time**: Building images on the host during the upgrade window adds latency to the upgrade process.
- **Build failures**: Build-time dependencies, disk space, or toolchain issues on any single host can cause that node's
  upgrade to fail.
- **No independent verification**: There is no mechanism for council members to independently verify that the software
  being deployed matches what was reviewed and approved.

The new model uses deterministic container images built using reproducible build processes. The image layer hashes are
identical regardless of where the image is built, enabling any council member to independently build and verify the
image. These pre-built images are hosted in container registries (up to 39, one per council member), providing
redundancy and independent verification.

### 3. Lack of Automated Host-Level Infrastructure Upgrades

The current UC operates as a host-level systemd service that manages Docker containers. It has no capability to upgrade
the host infrastructure itself — the operating system, container runtime, or any host-level tooling. Infrastructure
upgrades require separate, manual processes coordinated across all 39 node operators.

As the network migrates to a Kubernetes-based deployment model, the need for infrastructure upgrades expands to include
the Kubernetes cluster itself, the Kubernetes operator, host binaries, and the provisioning toolchain. The current
architecture provides no mechanism to coordinate these infrastructure upgrades as part of the network upgrade protocol.

This proposal introduces a layered upgrade architecture where a single upgrade manifest can trigger both consensus node
software upgrades and infrastructure upgrades, orchestrated through the existing freeze transaction protocol without
requiring changes to the consensus node code.

### 4. No Declarative State Management and Self-Healing Capabilities

Docker Compose provides basic container orchestration but lacks the self-healing, declarative state management, and
extensibility of Kubernetes. The current model requires the UC to imperatively manage container lifecycle — stopping,
rebuilding, and restarting containers through procedural scripts. There is no mechanism for the system to automatically
detect and recover from drift between desired and actual state.

The new model leverages Kubernetes Custom Resources (CRs) and a purpose-built operator to declaratively manage the
upgrade lifecycle. Each phase of the upgrade is represented as a CR with well-defined status conditions, providing a
clear audit trail and enabling the operator to reconcile toward the desired state automatically.

## Rationale

### Preserving the Freeze Transaction Protocol and Marker File Interface

A key design decision is to preserve the existing freeze transaction protocol (`PREPARE_UPGRADE`, `FREEZE_UPGRADE`,
`FREEZE_ABORT`) and the marker file interface (`execute_immediate.mf`, `freeze_scheduled.mf`, `now_frozen.mf`,
`freeze_aborted.mf`) without modification. This was chosen because:

- **Minimal consensus node changes**: The consensus node is the most security-critical component in the network.
  Minimizing changes to its codebase reduces the risk of introducing bugs that could affect network consensus or
  transaction processing.
- **Backward compatibility during migration**: During the transition period, some nodes may still run on the Docker
  Compose model while others migrate to Kubernetes. Preserving the marker file interface ensures both deployment models
  can coexist.
- **Proven protocol**: The freeze transaction protocol has been used successfully for many network upgrades. Its state
  machine semantics (including the irreversibility of the `FREEZING` state and the abort safety guarantees) are
  well-understood and battle-tested.

The UC sidecar acts as a translation layer — consuming the same marker files the current UC consumes, but producing
Kubernetes Custom Resources instead of directly managing containers.

### Separate CR Types

The upgrade lifecycle is modeled as four separate CR types (`NetworkUpgradePrepare`, `NetworkUpgradeFreeze`,
`NetworkUpgradeExecute`, `NetworkUpgradeFreezeAbort`) rather than a single CR with a workflow step enum. This was chosen
because:

- **Single responsibility**: Each CR type has a focused reconciler with clear inputs, outputs, and status conditions.
  This simplifies both the implementation and the debugging of individual upgrade phases.
- **Independent lifecycle**: Each CR is created by the UC when the corresponding marker file appears, reflecting the
  actual progression of the upgrade protocol. CRs are not created speculatively — they appear only when the consensus
  node has confirmed that phase.
- **Audit trail**: All CRs are retained after the upgrade completes, providing a record of the upgrade's progression
  through each phase, including timestamps, selected registries, and validation results. CRs are not deleted in the
  current implementation. On long-lived networks (5–10 years, 100+ upgrades), accumulation may become a concern for
  etcd storage and watch cache size; TTL-based cleanup of terminal-phase CRs by solo-operator is the intended future
  solution when that becomes necessary.
- **Observability**: Operators can query individual CR types to understand the state of any upgrade phase without
  parsing a complex nested status structure.

The alternative — a single `NetworkOperation` CR with a `workflowStep` field — was the previous design and proved to be
complex to manage, requiring careful state machine logic within a single reconciler and making it difficult to represent
the handoff between UC-created steps and operator-processed steps.

### UC Creates All CRs; Operator Only Reconciles

The UC sidecar is responsible for creating all upgrade-related CRs. The operator's reconcilers process these CRs but
never create the next CR in the chain. This separation was chosen because:

- **Source of truth alignment**: The consensus node is the authoritative source of upgrade protocol state (via marker
  files). The UC, which directly observes these marker files, is the natural component to translate protocol state into
  Kubernetes resources.
- **No speculative creation**: The operator never needs to guess whether the next phase will occur. For example, the
  `NetworkUpgradeFreeze` CR is only created when the CN actually writes `freeze_scheduled.mf`, not when the
  `NetworkUpgradePrepare` CR succeeds.
- **Clear failure boundaries**: If the UC fails, no new CRs are created. If the operator fails, existing CRs are not
  reconciled. These are independent failure modes with distinct recovery procedures.

### Persist-Before-Send Pattern for Crash Safety

The UC follows a strict persist-before-send pattern: it constructs each CR object, persists it to local disk, and only
then sends it to the Kubernetes API. This pattern ensures:

- **Crash recovery**: If the UC crashes after persisting but before sending, it can resume from the disk copy on
  restart. If it crashes after sending, the cluster already has the CR — the startup cluster query covers this case.
- **Idempotency**: On restart, the UC queries the cluster (source of truth) and scans persisted CRs on disk to
  determine what has already been completed, then resumes from wherever it left off.

### Infrastructure Upgrade via `solo-provisioner-daemon`

`solo-provisioner` is an opinionated CLI tool for provisioning and managing Hedera node infrastructure on bare-metal
machines; it co-ordinates with two long-running components: the `solo-provisioner-daemon` (host-level, outside
the cluster, started at install time) and the `solo-operator` Kubernetes operator. Full architecture is described in HIP
1495.

When the upgrade manifest includes a new `provisioner.cli.version` or `provisioner.daemon.version`, the daemon
handles the cluster teardown, reinstall, and any host-level tooling upgrades before setting
`PendingNodeUpgrade`. This is necessary because:

- **The cluster must be torn down**: Infrastructure upgrades may require reinstalling the Kubernetes cluster itself,
  which means etcd (and all CRs) will be destroyed. A component outside the cluster must orchestrate this.
- **Version coherence**: The solo-provisioner has built-in knowledge of which versions of Kubernetes, the operator, and
  host binaries are compatible. Delegating to it ensures the entire stack is upgraded coherently.
- **CR preservation**: The Kubernetes etcd data is stored on a hostPath volume that is never destroyed during
  teardown/reinstall, so all CRs survive automatically without any re-submission.

In the **Mainnet profile**, the Kubernetes operator (`solo-operator`) **always** routes through the daemon for every
upgrade — whether or not an infrastructure upgrade is required. This unified path simplifies the operator and ensures
all config file placement, infra checks, and `PendingNodeUpgrade` signaling happen in one consistent place.

In the **cluster-only profile** (fully Kubernetes-managed environments with no bare-metal machine access), the UC sidecar
absorbs this role: after the operator scales the CN pod to zero and sets `ReadyForProvisionerDaemon`, the UC detects the
status change and sets `PendingNodeUpgrade` directly. Infrastructure upgrades are not supported in this profile;
a manifest containing an `infrastructure` section is rejected with an explicit error by both the UC and the
operator. The `Orbit.Spec.Consensus.ProvisionerDaemonEnabled` field controls which path is taken (see HIP-1495). Both
paths are illustrated in detail in [§ Execute Phase Flow](#execute-phase-flow).

### Infrastructure and CN Upgrade in a Single Window

Every upgrade — whether it includes infra changes, CN changes, or both — is executed within a **single freeze
window**. A separate infra-only freeze window is not supported.

This is possible because the infra upgrade is performed by the `solo-provisioner-daemon` **before** the CN pod
is restarted. The daemon reads `manifests/infrastructure-versions.yaml`, applies any required infra changes
(Kubernetes version, CRI-O, host binaries), and only then signals the CN to come back up on the updated stack.
Infra and CN upgrades are sequenced within the same window rather than split across two freeze events.

The `manifests/infrastructure-versions.yaml` file also acts as a **safety gate**: the UC and daemon validate
that installed operator and daemon versions meet the declared minimums before the freeze proceeds. If the check
fails, the UC emits a `ManifestInvalid` event and the upgrade is blocked — the node operator must manually
upgrade the infra stack first, then re-run the upgrade.

### Polling Instead of Inotify for Marker File Detection

The UC uses polling (default: every 60 seconds) rather than inotify/fsnotify for detecting marker files. This was
chosen because:

- **hostPath reliability**: In Kubernetes, hostPath volumes can have inotify delivery issues depending on the container
  runtime and storage backend. Polling avoids this class of problems entirely.
- **Simplicity**: Polling is straightforward to implement, test, and debug. The 60-second interval is acceptable because
  marker files are written infrequently (days apart in the upgrade protocol), and the polling interval is configurable.
- **Restart resilience**: On restart, the poller naturally discovers all existing marker files and handles them in
  order, without needing to replay missed filesystem events.

### Manual Intervention for Failure Recovery

The system does not implement automatic rollback at any stage. If an upgrade fails (image pull failure, pod crash,
infrastructure rebuild failure), manual intervention by the node operator is required. This was chosen because:

- **Network-level coordination**: An automatic rollback on one node cannot account for the state of the other 38 nodes.
  The network can tolerate up to 1/3 of nodes being temporarily offline, so a failed node does not necessarily require
  rollback — it may just need retry or repair.
- **Safety over automation**: Automatic rollback of infrastructure (reverting a Kubernetes cluster version, for example)
  carries significant risk. A human operator can assess the situation and choose the safest recovery path.
- **Observability enables response**: The event system (JSONL files, Loki logs, K8s events, and exposed metrics) provides
  operators with the information needed to diagnose and resolve failures promptly, and a foundation to build their own
  dashboards and alert policies.

### Upgrade Manifest

The upgrade package contains machine-readable manifests and optional configuration files. The full package
directory structure and manifest schemas (`consensus-node-components.yaml`, `infrastructure-versions.yaml`,
`state-sources.yaml`, `external-files.yaml`) are defined in **HIP-1494 — Consensus Node Deployment Package
Specification**. This HIP uses those schemas without restating them.

Key behavioural points specific to the upgrade path:

- Upgrades are only supported from version N to N+1. Skipping versions (e.g., upgrading directly from N to N+2 or
  higher) is not supported. This ensures compatibility and simplifies both upgrade and rollback procedures.
- Each manifest file (`consensus-node-components.yaml`, `infrastructure-versions.yaml`, etc.) carries a
  `schemaVersion` field validated by the UC and daemon on receipt. Unknown versions cause a `ManifestInvalid`
  event and halt processing. Within a known `schemaVersion`, unknown fields are silently ignored (lenient
  parsing), allowing additive changes without requiring all components to upgrade simultaneously. Breaking
  changes require a `schemaVersion` bump — see HIP-1494 for the full schema evolution rules.
- The `registries` list in `consensus-node-components.yaml` can include up to 39 entries (one per council
  member). The operator validates all registries at prepare time and selects one randomly per component to
  distribute pull load and reduce correlated failure risk.
- The `layerHashes` field contains image layer hashes for each supported platform, used to verify image
  integrity before deployment.
- The `manifests/infrastructure-versions.yaml` file is read by the UC and daemon as a version gate before
  creating any CRs. If any installed infra component is below the declared minimum, the UC emits a
  `ManifestInvalid` event and refuses to proceed. The node operator must manually upgrade the infra stack first.
- Configuration files are **never embedded in any CR**. After the infra upgrade completes, the daemon scans
  `data/config/` in the extracted package and, for each recognised filename, creates the corresponding dedicated
  Kubernetes CR. The filename-to-CR-kind mapping is hardcoded in the daemon/UC. The operator's CR reconciler
  owns all domain-specific logic: ConfigMap name derivation, key mapping, and semantic content validation.
- `manifests/external-files.yaml` is optional. It lists files too large for CR delivery (recommended limit: 1
  MB). Files are downloaded by the UC or daemon at the phase declared in the `phase` field (`prepare` or
  `freeze`). The daemon confirms presence and hash at execute time — no download occurs unexpectedly during the
  freeze window.
- The upgrade package uses `data/config/` for ConsensusConfig files (delivered via dedicated CRs) and
  `manifests/infrastructure-versions.yaml` for InfraConfig (placed directly on the host filesystem by the
  daemon). All delivery happens during the execute phase before the new pod starts.
- Dedicated ConsensusConfig CRs are never deleted after creation — they form part of the immutable upgrade
  audit trail alongside the `NetworkUpgrade*` CRs. The `hedera.com/operation-id` label on each CR links it
  back to the upgrade that created it.
- **Idempotency**: if the daemon restarts mid-way, it checks whether a CR for a given filename already exists
  with a matching `hedera.com/operation-id` label. If so, it skips creation and proceeds to watch the existing
  CR's status. This ensures re-runs are safe without duplicate CRs or double-writes.
- The `operationId` is a unique identifier for the upgrade operation, generated deterministically by tooling at
  apply time. Its format is `upgrade-<YYYYMMDDTHHmmssZ>-<version>` — e.g. `upgrade-20260415T143000Z-v0.75.0` —
  where the timestamp is the UTC freeze time and the version is the consensus node package version. The UC
  applies this as a label on all CRs it creates, enabling easy correlation of CRs to specific upgrade operations.
- **CR naming**: The recommended convention is `upgrade-<freeze-time>-<step>` where `<freeze-time>` uses lowercase
  ISO 8601 to satisfy Kubernetes RFC 1123 name rules (`[a-z0-9-.]` only) — e.g. `upgrade-20260415t143000z-execute`.
  Any unique, valid K8s name is acceptable; `metadata.name` carries no semantic meaning. The `operationId` field
  in the spec (and the `hedera.com/operation-id` label) is the authoritative identity of the upgrade — all
  correlation across Prepare, Freeze, and Execute CRs is done via `operationId`, not the resource name.
- The format of `infrastructure-versions.yaml` is defined by the solo-provisioner. The operator does not need to
  understand its contents; the daemon can access and parse from the path when an infrastructure upgrade is needed.
- ConsensusConfig files are optional. If not present, the system continues to use defaults baked into the
  container image. The council can choose to include configuration overrides in some upgrades and not others,
  without requiring changes to the operator.

### Deterministic Container Images

Consensus node images are built using a deterministic Dockerfile that produces identical layer hashes regardless of the
build environment. The Dockerfile is located in the `hiero-consensus-node` repository at
`hedera-node/infrastructure/docker/containers/production-next/consensus-node/Dockerfile`.

- It is not mandatory for all 39 council members to build and host the image, as long as at least one does.
- All hosted images across registries must have matching layer hashes.
- Council members can independently build, verify hashes, and host images in their own registries for redundancy and
  independent verification.
- The operator verifies layer hashes from the manifest against the actual image before selecting a registry for
  deployment.

### Configuration File Handling

**ConsensusConfig vs InfraConfig:**

- **ConsensusConfig — Dedicated CR per file, ConfigMap-mounted** (`application.properties`, `throttles.json`,
  `feeSchedules.json`, `api-permission.properties`, `log4j2.xml`, `settings.txt`, `genesis-network.json`, etc.):
  During the execute phase, the daemon scans `data/config/` in the extracted package. For each recognised
  filename, it creates the corresponding dedicated Kubernetes CR. The filename-to-CR-kind mapping is hardcoded
  in the daemon/UC — no dispatch field in the package manifests. Each CR's reconciler owns all domain-specific
  logic: ConfigMap name derivation, key mapping, and semantic content validation. No config content is ever
  embedded directly in a `NetworkUpgrade*` CR, keeping etcd free of large or variable-size file contents.

  Example dedicated CRs created from `data/config/`:

  ```yaml
  apiVersion: hedera.com/v1alpha1
  kind: Log4j2Config          # CR kind for log4j2.xml
  metadata:
    name: <capsule-name>-log4j2
    labels:
      hedera.com/capsule: <capsule-name>
      hedera.com/operation-id: "upgrade-<freeze-time>-<version>"
  spec:
    orbit: <orbit-name>
    content: |
      <Configuration>...</Configuration>
  status:
    phase: Pending | Validated | Succeeded | Failed
  ---
  apiVersion: hedera.com/v1alpha1
  kind: ThrottleConfig         # CR kind for throttles.json
  metadata:
    name: <capsule-name>-throttles
    labels:
      hedera.com/capsule: <capsule-name>
      hedera.com/operation-id: "upgrade-<freeze-time>-<version>"
  spec:
    orbit: <orbit-name>
    content: |
      { "throttles": [...] }
  status:
    phase: Pending | Validated | Succeeded | Failed
  ```

  The operator's reconciler for each CR kind determines the target ConfigMap name and key, and performs any
  semantic validation appropriate for that file type before updating the ConfigMap.

- **InfraConfig — Daemon-placed, hostPath-mounted**: Files consumed by processes outside the cluster (e.g.
  `manifests/infrastructure-versions.yaml`) are placed atomically by the daemon at the resolved host path
  during the execute phase. They are not delivered via ConfigMap or any CR.

The correct configuration is always in place before the new pod starts — ConsensusConfig via dedicated CRs →
ConfigMaps, InfraConfig via direct daemon placement on the host filesystem.

### Freeze Transaction Protocol

The existing freeze transaction protocol is **unchanged**. The same three transaction types (`PREPARE_UPGRADE`,
`FREEZE_UPGRADE`, `FREEZE_ABORT`) and the same marker file interface are preserved. The consensus node code requires
minimal changes — it continues to extract the archive and write marker files exactly as before.

The only difference is what the archive contains: a small manifest and optional config files instead of JAR files and
lifecycle scripts.

### Upgrade Controller (UC) Sidecar

The UC is reimplemented as a Go binary (`solo-operator/cmd/uc`) running as a sidecar container in the same pod as the
consensus node. It is released as a separate Docker image on GitHub Package Registry with its own release cadence.

**Configuration** (all configurable via flags, environment variables, or ConfigMap):

**Operation-id Deduplication:**
Deduplication is enforced at the Kubernetes API level — before creating any CR, the UC queries the cluster for an
existing CR with the same `hedera.com/operation-id` label. If one already exists, creation is skipped. This is the
authoritative guard and is crash-safe: a UC restart after a partial operation will re-query and find the existing CR
rather than creating a duplicate.

An in-memory CR state map is maintained as a performance cache to avoid a K8s API round-trip on every poll tick. It
is populated at startup by a query on `hedera.com/operation-id`, and invalidated (triggering a fresh query) on any
K8s API error. The map is a cache only — correctness depends on the API-level check, not on the map's accuracy.

| Parameter           | Default                                                       | Description                                                                                                                                                                                                                                                                 |
|---------------------|---------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `--marker-path`     | `/opt/hgcapp/services-hedera/HapiApp2.0/data/upgrade/current` | Directory to poll for marker files                                                                                                                                                                                                                                          |
| `--events-path`     | `/opt/solo/weaver/uc/events/`                                 | JSONL event log directory                                                                                                                                                                                                                                                   |
| `--cr-persist-path` | `/opt/solo/weaver/uc/crs`                                     | Directory for persisted CR YAML files                                                                                                                                                                                                                                       |
| `--backup-path`     | `/opt/solo/weaver/uc/backup/`                                 | Pre-upgrade backup directory                                                                                                                                                                                                                                                |
| `--poll-interval`   | `60s`                                                         | Marker file polling interval. Automatically drops to **1s** after `freeze_scheduled.mf` is detected (to minimise `now_frozen.mf` detection latency during the freeze window). Reverts to the configured interval after `now_frozen.mf` or `freeze_aborted.mf` is processed. |

**RBAC**: The UC pod uses a dedicated ServiceAccount with permissions scoped to creating and reading the four upgrade
CR types only.

**Marker File Processing Lifecycle**:

For each marker file, the UC follows this sequence:

1. **Detect** — Polling discovers a new marker file.
2. **Construct** — Build the corresponding CR object by parsing the package manifests (`consensus-node-components.yaml`,
   etc.) and reading relevant state (e.g., Prepare CR status for registry selection).
3. **Persist** — Write the CR as YAML to local disk (`--cr-persist-path`). This happens **before** any network
   communication, ensuring crash safety.
4. **Send** — Deliver the CR to its destination (always the Kubernetes API):
    - For `execute_immediate.mf`, `freeze_scheduled.mf`, `freeze_aborted.mf`, `now_frozen.mf`: send to Kubernetes API.
    - The operator always handles `now_frozen.mf` by scaling the pod down and setting CR status to
      `ReadyForProvisionerDaemon`.
      In the **Mainnet profile**, the solo-provisioner daemon then sets `PendingNodeUpgrade`. In the **cluster-only
      profile**, the UC sidecar detects `ReadyForProvisionerDaemon` and sets `PendingNodeUpgrade` directly (no daemon
      present).
5. **Update in-memory map** — On K8s API success, update the in-memory CR state map (`prepare: created`,
   `freeze: created`, etc.). The marker file stays on disk until final cleanup. The map entry is what
   prevents the poll loop from re-processing the marker on the next tick.
6. **Monitor** — A background coroutine polls the Kubernetes API to track CR status for all active upgrade CRs.
7. **Cleanup** — When the upgrade completes (all CRs reach `Succeeded`) or is aborted (`FreezeAbort` CR processed),
   clean up all marker files, persisted CRs on disk, and upgrade artifacts. **On startup**, the UC queries the cluster
   for any previous operation-id CRs. If a previous Execute CR has reached `Succeeded` or `Failed` and disk artifacts
   remain, the UC performs cleanup immediately before entering its normal polling loop. This ensures no leftover
   artifacts from previous operations interfere with future upgrades.

   **Ordering constraint**: The UC must **not** delete the upgrade directory (`--marker-path`) until after the daemon
   has set the Execute CR status to `PendingNodeUpgrade`. The daemon reads package manifests and config files from this
   directory during the execute phase. The upgrade directory is a hostPath that persists across pod restarts, but the
   UC's cleanup coroutine must check for `PendingNodeUpgrade` (or `Succeeded`) status before removing it.

**Idempotent Recovery on Restart**:

When the UC starts (or restarts), it determines what has already been completed by checking two sources in priority
order:

1. **Kubernetes cluster** (source of truth) — queries for existing CRs by `hedera.com/operation-id` label, where the
   `operation-id` is derived deterministically by the tooling at apply time (e.g. from the freeze timestamp and
   package version); it is not embedded in the package by the council.
2. **Persisted CRs on disk** — checks `--cr-persist-path` for CRs written to disk but not yet delivered to the cluster
   (crash window between persist and send).

If a CR exists in the cluster, the UC considers it delivered and skips to monitoring its status. If a CR exists on disk
but not in the cluster, it re-sends it. If neither exists, it checks for the marker file and processes it from the
beginning.

The startup cluster query also populates the **in-memory CR state map** used as a performance cache by the poll
loop (see Operation-id Deduplication above). Because marker files persist until final cleanup, the poll loop sees
the same marker files on every tick for the duration of the upgrade — the cache avoids redundant K8s API calls on
each tick. The cache is updated on K8s API success and invalidated on any API error, triggering a fresh cluster
query before the next action.

**Marker-to-CR Mapping**:

| Marker File            | CR Type                     | Destination    | On success                                                                                                                                                                                                                                                                                |
|------------------------|-----------------------------|----------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `execute_immediate.mf` | `NetworkUpgradePrepare`     | Kubernetes API | Update in-memory map (`prepare: created`). Marker stays until final cleanup.                                                                                                                                                                                                              |
| `freeze_scheduled.mf`  | `NetworkUpgradeFreeze`      | Kubernetes API | Update in-memory map (`freeze: created`). Marker stays until final cleanup.                                                                                                                                                                                                               |
| `now_frozen.mf`        | `NetworkUpgradeExecute`     | Kubernetes API | Update in-memory map (`execute: created`). Marker stays until final cleanup. Operator scales pod down, sets `ReadyForProvisionerDaemon`. Mainnet: daemon moves config files, runs infra upgrades, sets `PendingNodeUpgrade`. cluster-only: UC coroutine sets `PendingNodeUpgrade` directly. |
| `freeze_aborted.mf`    | `NetworkUpgradeFreezeAbort` | Kubernetes API | Update in-memory map (`abort: created`). ### Custom Resource Definitions

All upgrade CRs share the following metadata:

```yaml
metadata:
  labels:
    hedera.com/operation-id: "upgrade-<freeze-time>-<version>"  # Links all CRs for one upgrade
    hedera.com/step: "prepare" # "prepare" | "freeze" | "execute" | "abort"
  annotations:
    hedera.com/created-by: "UC" # "UC" | "SOLO_PROVISIONER_UPGRADE"
```

CRs are retained after an upgrade completes. Reconcilers check status conditions to determine whether a CR has been
processed and skip re-processing accordingly. See [CR Retention Policy](#networkupgradeexecute-cr-retention-policy)
for the accumulation bounds and deletion ownership.

**Ordering enforcement**: Each reconciler queries for CRs with the same `hedera.com/operation-id` label and verifies
that all prerequisite CRs (earlier steps) have reached `Succeeded` phase before processing the current CR.

#### NetworkUpgradePrepare

Created by UC when `execute_immediate.mf` is detected.

```yaml
apiVersion: hedera.com/v1alpha1

metadata:
  name: upgrade-<freeze-time-lowercase>-prepare
  labels:
    hedera.com/operation-id: "upgrade-<freeze-time>-<version>"

**PrepareReconciler behavior**:

1. Validates the manifest schema and `operationId` uniqueness.
2. Iterates through the `registries` list for each component **present** in the CR `images` section with `enabled: true`, in **random order** per component to distribute load. Components omitted from `images` entirely are skipped — their running image is preserved unchanged. Components present but with `enabled: false` are also skipped for registry validation — they will be removed from the pod during execute.
3. For each registry entry, attempts to pull and verifies layer hashes against the component's `layerHashes` for the host platform. **Every registry is checked** — the reconciler does not stop at the first valid one. Each result (`Valid` / `Unreachable` / `HashMismatch`) is recorded in the CR status `selectedSources` per component.
4. Emits a `RegistryAuditComplete` event summarising reachability per component (e.g., `"consensusNode: 2/2 registries reachable"`). This gives node operators a cross-node view of which registries are accessible from each host — a node that cannot reach any registry at prepare time must be remediated before the freeze window.
5. From the valid registries, **selects one at random** per present-and-enabled component and records it as the `selectedSources` entry (with `image`, `digest`, and `layerHashesVerified`). If no valid registry exists for any such component, the reconciler marks the CR phase as `Failed` and emits a `RegistryInvalid` event — the upgrade cannot proceed until registry access is restored.
6. Updates CR status (`selectedSources`) with the selected registry and digest **per present-and-enabled component**. Omitted components and `enabled: false` components produce no `selectedSources` entry.
7. Marks the CR phase as `Succeeded`.

**ExecuteReconciler behavior** (unified single path for all upgrades):

Each `NetworkUpgradeExecute` CR is scoped to a single `ConsensusCapsule` via the required `spec.consensusNode`
field. In the single-node-per-cluster topology there is exactly one such CR per operation. In multi-capsule
clusters, the UC creates one Execute CR per capsule, all sharing the same `operation-id`, and the reconciler
coordinates across these siblings to guarantee a globally consistent patch order (see **Multi-capsule
coordination** below).

1. Verifies that `NetworkUpgradePrepare` and `NetworkUpgradeFreeze` CRs with the same `operation-id` have reached
   `Succeeded`.
2. **Re-validates the selected registry** at the start of processing, before patching the `ConsensusCapsule`. If the previously selected registry for any component is unreachable, the reconciler iterates through the full `registries` list for that component in the `images` section of the Execute CR spec (self-contained, copied from the Prepare CR by the UC). If needed, the reconciler may also query the `NetworkUpgradePrepare` CR by `operation-id` for the authoritative `images` section. The reconciler selects the first valid one, updates the Execute CR `selectedSources` status with the new selection (emitting a `RegistryFailover` warning event), and proceeds. Only if no valid registry can be found for any component does the reconciler set the CR to `Failed`.
3. **Scales down the CN pod** by setting `ConsensusCapsule.spec.offline = true`. The reconciler operates at the
   CR level — it never manipulates StatefulSet replicas directly; the `ConsensusCapsuleReconciler` observes the
   `offline` flag and scales the underlying StatefulSet to `0`. Since stream uploaders run as sidecar containers
   within the CN pod, scaling to `0` stops all uploaders automatically — no separate scaling step is needed. The
   CN was in `FREEZE_COMPLETE` and is safe to terminate. After the capsule has been patched (see step 6), the
   reconciler clears `offline` (`offline = false`) to bring the upgraded pod back up.

   **Idempotency**: If the reconciler detects that the capsule is already `offline` and the CR status is already
   `ReadyForProvisionerDaemon` (e.g., after a cluster rebuild where the new operator picks up an in-progress upgrade),
   it skips the
   scale-down step entirely and proceeds to watch for `PendingNodeUpgrade`. The reconciler never sets the status back to
   `ReadyForProvisionerDaemon` if it is already set.

4. Sets the CR status to `ReadyForProvisionerDaemon`. The actor that takes over next depends on the deployment profile:
    - **Mainnet profile**: The solo-provisioner daemon (continuously monitoring the Kubernetes API) detects the status
      change and takes over.
    - **cluster-only profile**: The UC sidecar (running in the pod before it was scaled down, or in a new pod after
      cluster restart) detects `ReadyForProvisionerDaemon` and sets `PendingNodeUpgrade` directly.
5. *(Mainnet profile)* The daemon performs the following steps in order:
    1. **InfraConfig placement**: moves `manifests/infrastructure-versions.yaml` and any other InfraConfig
       files from the upgrade directory to their correct host filesystem locations (atomic write: `.tmp` then
       rename).
    2. **Infra upgrade** (if needed): reads `infrastructure-versions.yaml` to determine whether an infra
       upgrade is required; sets CR status to `PendingInfraUpgrade`, runs infrastructure upgrades (cluster
       teardown/reinstall, operator upgrade, daemon self-upgrade if required — see self-upgrade protocol),
       then resumes. If no infra upgrade is needed, this step is skipped.
    3. **ConsensusConfig CR creation**: scans `data/config/` and, for each recognised filename, creates the
       corresponding dedicated Kubernetes CR. The operator reconciles each CR — applying domain-specific
       validation — and creates or updates the corresponding ConfigMap mounted into the CN pod.
    4. **Wait for reconciliation**: the daemon watches each config CR's status until all reach `Succeeded`.
    5. **Signal readiness**: sets the `NetworkUpgradeExecute` CR status to `PendingNodeUpgrade`.

   *(cluster-only profile)* No InfraConfig file placement or infra upgrade occurs. The UC sidecar detects
   `ReadyForProvisionerDaemon`, scans `data/config/` in the upgrade PVC and creates the appropriate dedicated
   CRs per file, waits for operator reconciliation, then sets `PendingNodeUpgrade` directly.
6. The reconciler detects `PendingNodeUpgrade` and **patches the `ConsensusCapsule` CR** according to three rules per component:
   - **Present and `enabled: true`** — update to the `image` and `digest` recorded in `selectedSources`.
   - **Omitted from `images`** — leave the current image unchanged; no patch is applied for this component.
   - **Present and `enabled: false`** — remove the sidecar container from the `ConsensusCapsule` spec; the `ConsensusCapsuleReconciler` drops it from the StatefulSet pod template on the next reconcile.

   This applies to `consensusNode` and all sidecars (`recordStreamUploader`, `eventStreamUploader`,
   `blockStreamUploader`, `backupUploader`, `uc`). The `ConsensusCapsuleReconciler` detects this patch and updates the
   underlying StatefulSet spec accordingly. Once every sibling capsule for the operation has been patched, the reconciler
   clears `offline` (see **Multi-capsule coordination**), and the `ConsensusCapsuleReconciler` scales the StatefulSet
   back to `1`, triggering a pod replacement with the upgraded images.
7. **Confirms the node reached `ACTIVE` and updates Execute CR**: The upgrade is marked `Succeeded` only once the
   consensus node reports `ACTIVE` platform status — **Kubernetes pod readiness alone is not sufficient**, because a
   `Ready` pod (container running, readiness probe passing) may still be replaying events, reconnecting, or otherwise
   short of `ACTIVE`. Confirmation is a single-writer handoff between the UC sidecar (which polls) and the reconciler
   (which owns `status.phase`):

   1. **Reconciler patches and signals.** Having patched the images and brought the capsule back online, the reconciler
      sets the `CapsulesOnline` condition while leaving the phase at `PendingNodeUpgrade`. That phase, together with
      `CapsulesOnline`, *is* the "begin polling" signal to the UC — no empty/intermediate phase is introduced.
   2. **UC polls for `ACTIVE`.** Observing `PendingNodeUpgrade` + `CapsulesOnline` on the CR (matched by `operation-id`),
      the UC sidecar polls `localhost:9999/metrics` for `platform_PlatformStatus` in a time-bounded loop. On `ACTIVE`
      (`= 2`) it sets the `NodeStarting` condition to `True`; on `CATASTROPHIC_FAILURE` (`= 11`) or its own timeout it
      sets `NodeStarting` to `False` with a reason. The UC **never writes `status.phase`**.
   3. **Reconciler finalizes.** Watching the CR, the reconciler reads the `NodeStarting` condition: `True` → it sets
      `status.phase = Succeeded`; `False` → `status.phase = Failed` with an `UpgradeFailed` event.
   4. **`Failed` requires operator intervention.** A `Failed` Execute CR is terminal — the reconciler stops requeuing
      and performs no automatic rollback. A node operator investigates the node and decides how to proceed.

   **Timeout (default 10 minutes), dual-owned.** Both the UC poll loop and a reconciler **backstop** are anchored to
   `status.nodeUpgradeStartTime` (the moment the CR entered `PendingNodeUpgrade`, not when it was created), so the bound
   measures only the node-restart window. The UC owns the primary wall-clock bound and writes a `False` `NodeStarting`
   condition when it expires; the reconciler backstop exists so that if the UC sidecar itself is dead and never writes
   any condition, the CR still transitions to `Failed` rather than hanging.

   **Authoritative platform-status signal**: the consensus node exposes a Prometheus metrics endpoint on port `9999`;
   the `platform_PlatformStatus` gauge reports the current platform status code (`ACTIVE` = 2, `FREEZE_COMPLETE` = 6,
   `CATASTROPHIC_FAILURE` = 11, etc.). This metric — not consensus-node log scraping — is the authoritative readiness
   signal: it is an exact numeric check, distinguishes `CATASTROPHIC_FAILURE` (a hard failure that aborts the wait),
   and is unaffected by log noise or rotation.

   **Single writer of terminal status**: `ACTIVE` is an *input* to `Succeeded`, never a confirmation applied *after* it.
   The reconciler is the **sole writer** of `status.phase`; the UC only writes the `NodeStarting` *condition*. After the
   reconciler sets the terminal phase, the UC observes it (by `operation-id`): on `Succeeded` it proceeds with artifact
   cleanup; on `Failed` it emits a warning event. It does not re-write the phase — `Succeeded` is the single, terminal
   completion state (there is no separate `Completed` phase). The UC thus has two ordered, distinct interactions with
   the CR: it writes the `NodeStarting` *input* condition before success, and reads the *terminal* phase after success
   to trigger cleanup — consistent with the overall design where all coordination flows through CR status fields.

   **Detection hook (active polling, not a probe).** The platform-status check is a general node-lifecycle primitive,
   not specific to upgrades — the same `platform_PlatformStatus` signal gates normal `ConsensusCapsule` start/restart
   (target `ACTIVE`) and freeze confirmation (target `FREEZE_COMPLETE`). Detection is performed by **actively polling**
   the Prometheus endpoint in a bounded retry loop (poll `http://localhost:9999/metrics`, parse the
   `platform_PlatformStatus` line, compare the numeric code to the target, and fail fast on `CATASTROPHIC_FAILURE`).
   The UC sidecar — co-located in the CN pod, so it reads `localhost:9999` directly with no pod `exec` — owns this
   poll and publishes the result as a CR condition. A Kubernetes **readiness/liveness probe is deliberately not used**:
   tying `ReadyReplicas` to `ACTIVE` would conflict with deliberate node start/stop and freeze operations, pulling a
   healthy-but-intentionally-not-`ACTIVE` node out of its Service endpoints. This mirrors `hiero-ledger/solo`, whose
   `checkNetworkNodeActiveness` polls the same `:9999` metric for every start, stop, and restart rather than relying on
   pod readiness.

**Multi-capsule coordination**

When a single cluster hosts more than one `ConsensusCapsule`, the UC creates one `NetworkUpgradeExecute` CR per
capsule, all sharing the same `operation-id`. Each reconciler instance owns its own capsule but coordinates with its
siblings (discovered by listing Execute CRs with the matching `operation-id`) so that the cluster-local capsules
transition as a group. Two barriers enforce a globally consistent order:

- **All offline before any patch**: before a reconciler patches its capsule's images, it verifies that *every*
  sibling capsule for the operation is `offline`. No capsule is patched until all are offline. This prevents a mixed
  state where some capsules have already restarted on the new image while others are still running the old one.
- **All patched before any online**: after patching, a reconciler verifies that *every* sibling Execute CR has its
  `ConsensusCapsulePatched` condition set to `True` before it clears `offline` on its own capsule. All capsules are
  brought back online together, on the new images.

In the single-node-per-cluster topology these barriers are trivially satisfied (one capsule, one CR) and impose no
additional waiting. The barriers operate only within a cluster; cross-cluster synchronization across the 39 council
nodes is still achieved exclusively through the Hedera consensus freeze protocol, not through Kubernetes.

   │          │          │           │ manifest    │
   │          │          │           │             │
   │          │          │           │ [if infra upgrade needed]
   │          │          │           │──fork──────>│
   │          │          │           │             │ stop daemon
   │          │          │           │             │ [if solo-provisioner
   │          │          │           │             │  self-upgrade needed:
   │          │          │           │             │  see self-upgrade protocol]
   │          │          │           │             │ teardown cluster
   │          │          │           │             │ (etcd preserved
   │          │          │           │             │  on hostPath)
   │          │          │           │             │ reinstall
   │          │          │           │             │ restore etcd
   │          │          │           │             │ deploy operator
   │          │          │           │             │ restart daemon
   │          │          │           │             │
   │          │          │           │ (restarted) │
   │          │          │           │ set status  │
   │          │          │           │ PendingNode─>
   │          │          │           │ Upgrade     │
   │          │          │           │             │
   │          │          │ [if no infra upgrade needed]
   │          │          │           │ set status  │
   │          │          │           │ PendingNode │
   │          │          │           │ Upgrade     │
   │          │          │           │             │
   │          │          │<──PendingNodeUpgrade────│
   │          │          │ patch     │             │
   │          │          │ Capsule   │             │
   │          │          │ new image │             │
   │          │          │ (no CM    │             │
   │          │          │  mounts   │             │
   │          │          │  needed)  │             │
   │          │          │           │             │
   │ ACTIVE   │(new UC) │ CR:       │             │
   │(upgraded)│sees CRs  │ Succeeded │             │
   │          │does      │           │             │
   │          │nothing   │           │             │
```

**Steps in detail**:

1. **UC detects `now_frozen.mf`** and persists the `NetworkUpgradeExecute` CR to disk, then creates it in the
   Kubernetes API. Initial status: `Pending`.
2. **UC updates its in-memory map** (`execute: created`) after the CR is confirmed in the cluster.
   `now_frozen.mf` stays on disk until final cleanup.
3. **The operator's `ExecuteReconciler`** verifies that `NetworkUpgradePrepare` and `NetworkUpgradeFreeze` CRs have
   reached `Succeeded`, then **scales down the CN pod** by setting the `ConsensusCapsule` StatefulSet replicas to `0`
   (stream uploaders are sidecars in the same pod and stop automatically — the CN was in `FREEZE_COMPLETE` and is safe
   to terminate), then sets the CR status to `ReadyForProvisionerDaemon`.
4. **The solo-provisioner daemon** (host process, continuously monitoring the Kubernetes API) detects the status change
   to `ReadyForProvisionerDaemon`. It moves necessary configuration files from the upgrade directory to their correct
   host
   filesystem locations. (The daemon accesses the upgrade directory at the well-known host path
   `/opt/hgcapp/services-hedera/HapiApp2.0/data/upgrade/current/` directly, since it runs as a host process with full
   filesystem access — no additional configuration of this path is required.) It then reads the manifest to determine if
   any infrastructure component upgrades are needed.
    - **If no infra upgrade needed**: sets CR status to `PendingNodeUpgrade` immediately.
    - **If infra upgrade needed**: sets CR status to `PendingInfraUpgrade`, runs the necessary infrastructure upgrades (
      cluster teardown/reinstall, host tooling — see self-upgrade protocol below if solo-provisioner itself needs
      upgrading), then sets CR status to `PendingNodeUpgrade`.
5. **The operator's `ExecuteReconciler`** detects `PendingNodeUpgrade` and **patches the `ConsensusCapsule` CR** with
   the new software versions: the CN container image, the UC sidecar image, and each stream uploader sidecar image —
   using the selected registry per component from `selectedSources` in the Execute CR status. The StatefulSet is
   scaled back to `1`, starting the upgraded pod with all sidecars.
6. **The new pod starts** with the upgraded CN and UC. The operator monitors until the CN reports `ACTIVE` platform
   status (confirmed via the UC sidecar), then marks the CR as `Succeeded`. If `ACTIVE` is not reached within the
   configurable timeout (default: 10 minutes), the CR is marked `Failed` and an `UpgradeFailed` event is emitted.

### `build/jobs/` Execution During Upgrade

The deployment package may include a `build/jobs/` folder with phase-scoped K8s Job descriptors (see HIP-1494 for the full schema). During an upgrade, the triggerer (daemon or UC, depending on phase) executes jobs with `context: upgrade` at the appropriate point in the upgrade state machine. Descriptors with `context: install` are silently skipped.

**Upgrade phase mapping:**

| `phase.name` | Triggerer | When |
|---|---|---|
| `prepare` | UC sidecar | After `execute_immediate.mf` detected; before the freeze window; CN still active |
| `frozen` | Daemon (Mainnet) / UC (cluster-only) | After `ReadyForProvisionerDaemon` is set; CN pod is stopped |
| `post-upgrade` | UC sidecar (new instance) | After CN pod restarts with new image and all containers report `Ready`; Execute CR reaches `Succeeded` |

**Execution model:**

1. The triggerer reads all descriptors from `build/jobs/`, filters to `context: upgrade` and the current `phase.name`, sorts by `phase.sequence` ascending.
2. For each job in order, before launching, the triggerer queries:
   ```
   kubectl get jobs -l hedera.com/operation-id=<id>,hedera.com/job-file=<stem>
   ```
   - Not found → inject labels, create Job, watch until terminal.
   - Found + succeeded → skip (idempotent).
   - Found + running → re-attach and watch.
   - Found + failed → delete and re-launch.
3. The triggerer injects two labels onto the Job manifest at apply time:
   - `hedera.com/operation-id: <operationId>` — links the Job to this upgrade operation.
   - `hedera.com/job-file: <descriptor-filename-stem>` — enables per-job idempotency lookup.
4. On completion: if `failurePolicy: Abort` and the pod exits non-zero, the upgrade halts and a `JobFailed` event is emitted. If `failurePolicy: Ignore`, the failure is logged and the next job proceeds.
5. Once all jobs for the phase complete (or are skipped), the triggerer continues with the next step in the upgrade flow.

**Post-rebuild daemon K8s connectivity**: After a cluster rebuild, the daemon retries all Kubernetes API connections
with exponential backoff until the operator has fully deployed the necessary RBAC resources (service accounts, roles,
and role bindings). Only once the RBAC is in place does the daemon proceed to set the `PendingNodeUpgrade` status.

**Cluster state persistence**: The Kubernetes etcd data is stored on a hostPath volume (`/opt/solo/weaver/sandbox/`)that
is **never destroyed** during a cluster teardown/reinstall. When solo-provisioner reinstalls the cluster, it restores
from this persisted etcd data. This means all CRs — both historical and the current upgrade's Prepare, Freeze, and
Execute CRs — survive the infrastructure upgrade automatically. The daemon never needs to download, re-submit, or
reconstruct CRs.

### Solo-Provisioner Self-Upgrade Protocol

When the daemon reads the manifest and finds that `provisioner.cli.version` or `provisioner.daemon.version`
differs from the currently installed versions, it follows this self-upgrade protocol. If either version changed,
both binaries are upgraded together (paired atomic upgrade avoids partial-state issues).

**Responsibility split**: The daemon is responsible only for downloading and verifying the new CLI binary. The
CLI (`provisioner self upgrade`) owns the remainder of the protocol — including downloading, validating, and
installing the new daemon binary. This mirrors how the CLI validates all other infrastructure component versions
(K8s, operator, host binaries): the daemon binary version in `infrastructure-versions.yaml` is just another
infra component the CLI is responsible for. The CLI may reject the specified daemon version as incompatible,
aborting with `SelfUpgradeFailed` before any installation occurs.

0. **Write self-upgrade state file** to `/opt/solo/weaver/daemon/self-upgrade.yaml`. The file captures the
   operation metadata plus the full `provisioner` block copied verbatim from `infrastructure-versions.yaml`
   (versions, algorithms, and checksums), so that the detached child and recovery tooling have everything they
   need without re-reading the original manifest:
   ```yaml
   operationId: "upgrade-20260415T143000Z-v0.75.0"
   startedAt: "<RFC3339 timestamp>"
   status: "in-progress"
   manifestPath: "/opt/hgcapp/services-hedera/HapiApp2.0/data/upgrade/current/manifests/infrastructure-versions.yaml"
   provisioner:
     cli:
       version: "0.42.0"
       algorithm: "sha256"
       checksum: "abc123..."
     daemon:
       version: "0.33.0"
       algorithm: "sha256"
       checksum: "def456..."
   ```
   This file is the primary diagnostic artifact if the node is found with no daemon running and no cluster
   accessible — it tells operators exactly what was being upgraded and whether the upgrade completed.

1. **Download and verify the new CLI binary** to a temp location:

   a. Download `solo-provisioner` (CLI) using `provisioner.cli.version`:
      ```
      /tmp/solo-provisioner-v<cli-version>
      ```
   b. Verify against `provisioner.cli.algorithm` + `provisioner.cli.checksum`. Abort with `SelfUpgradeFailed` on
   mismatch.

   The daemon retries the download with exponential backoff (up to a configurable maximum number of attempts). If all
   retries are exhausted, the daemon sets the Execute CR status to `Failed` with a detailed reason (e.g.,
   `DownloadFailed: <url> unreachable after N attempts`) and emits a `SelfUpgradeFailed` event. No further action is
   taken — manual intervention is required.

2. **Start a new detached child process** that survives daemon death. The old daemon spawns the newly downloaded
   `solo-provisioner` binary via Go's `os/exec`, with `SysProcAttr.Setsid = true` so the child runs in its own
   session and is not killed when the old daemon is stopped in step 3f below.
   The child executes:
   ```
   /tmp/solo-provisioner-v<cli-version> provisioner self upgrade \
     --state-file=/opt/solo/weaver/daemon/self-upgrade.yaml
   ```
   The state file contains the manifest path and the full `provisioner` block — the detached child reads it
   directly rather than requiring all version and checksum details to be passed as flags.

3. **The detached child** (`provisioner self upgrade` command) then executes sequentially:

   a. **Read the state file** to obtain the manifest path, target daemon version, and checksum. Read
   `infrastructure-versions.yaml` at `manifestPath` to confirm values are consistent.

   b. **Validate all infra version compatibility** — the CLI applies the same compatibility matrix it uses for
   all infrastructure components (K8s, operator, host binaries, and daemon). For each component declared in
   `infrastructure-versions.yaml`, the CLI checks the specified version against its built-in supported range.

   **On incompatibility — default behaviour (hard stop)**:
   - Emit a structured error to journald listing each incompatible component, the version specified, and the
     range this CLI release accepts. Example:
     ```
     version incompatible: provisioner.daemon.version=0.29.0 (accepted: >=0.31.0); \
                           infrastructure.k8s.version=1.28.0 (accepted: >=1.29.0)
     ```
   - Emit a `VersionIncompatible` K8s Warning event with the same detail.
   - Write a `VersionIncompatible` JSONL entry to the upgrade event log.
   - Set Execute CR status to `Failed` with reason `VersionIncompatible`.
   - **Abort with `SelfUpgradeFailed` before any files are modified** — no partial upgrades.

   **Operator escape hatch (`--override-version-check`)**:
   If the node operator must proceed despite the incompatibility (e.g., council-coordinated emergency recovery),
   the command can be re-run manually with the override flag:
   ```
   provisioner self upgrade \
     --state-file=/opt/solo/weaver/daemon/self-upgrade.yaml \
     --override-version-check
   ```
   When this flag is present, the CLI:
   - Logs a prominent `WARNING: version compatibility check bypassed by operator` to journald, K8s event
     (Warning), and JSONL — so there is always a durable record of the override.
   - **Does not suppress checksum verification** — integrity checks are security controls and are never
     bypassable.
   - Proceeds with the remainder of the protocol at the operator's risk.

   This flag is not part of the normal automated upgrade path. It is an emergency tool for node operators only.

   c. **Download and verify the new daemon binary** to a temp location:
      ```
      /tmp/solo-provisioner-daemon-v<daemon-version>
      ```
      Verify against `provisioner.daemon.algorithm` + `provisioner.daemon.checksum`. Abort with
      `SelfUpgradeFailed` on mismatch. Retries with exponential backoff, same policy as step 1.

   d. **Archive old binaries** for quick recovery (uses the `operationId` from the state file to name backups):
      ```
      cp /opt/solo/weaver/bin/solo-provisioner        /opt/solo/weaver/bin/solo-provisioner-<operation-id>.bak
      cp /opt/solo/weaver/bin/solo-provisioner-daemon /opt/solo/weaver/bin/solo-provisioner-daemon-<operation-id>.bak
      ```
   e. **Atomically install** new binaries (temp file + rename for crash safety):
      ```
      install -m 0755 /tmp/solo-provisioner-v<cli-version>        /opt/solo/weaver/bin/solo-provisioner
      install -m 0755 /tmp/solo-provisioner-daemon-v<daemon-version> /opt/solo/weaver/bin/solo-provisioner-daemon
      ```
   f. **Stop old daemon**: `systemctl stop solo-provisioner`
   (the detached `provisioner self upgrade` child calls systemd to stop the old daemon process that spawned it;
   the detached child is in its own session and is unaffected)
   g. **Start new daemon**: `systemctl start solo-provisioner`
   (again called by `provisioner self upgrade` — systemd is the mechanism, but the lifecycle is owned by the
   self-upgrade command, not the operator or systemd directly)

   h. **Health-check loop**: Poll `GET /health` on the daemon's Unix domain socket at
   `/opt/solo/weaver/daemon/daemon.sock`, retrying for up to a configurable timeout (default: 120 seconds).

   i. **If daemon is healthy**: The detached child process updates the state file to
   `status: succeeded, completedAt: <timestamp>`,
   cleans up the backup files (`solo-provisioner-<operation-id>.bak` and `solo-provisioner-daemon-<operation-id>.bak`),
   and exits 0. The new daemon is now in control and proceeds to set `PendingNodeUpgrade`.

   j. **If daemon is not healthy** after timeout — the detached child process automatically attempts recovery:
    - **Automatic recovery**: restore both binaries from `.bak` files → `systemctl start solo-provisioner` → re-run
      health-check loop. If daemon comes up, it sets the Execute CR `DaemonResult` condition to `Failed` (reason:
      `SelfUpgradeFailed`) and sets status to `PendingNodeUpgrade` so the ExecuteReconciler can mark the CR `Failed`.
      The state file is updated to `status: recovery-succeeded`.
    - **If automatic recovery also fails**: the detached child process updates the state file to `status: failed` and
      exits
      non-zero. The state file is the primary diagnostic artifact — the node operator should inspect it first. The node
      operator must then manually run `solo-provisioner consensus node upgrade-recover`, which performs the same
      restore-and-restart sequence but with interactive diagnostics. If all recovery attempts fail, the node operator
      runs `solo-provisioner consensus node upgrade-diagnose` to collect artifacts for escalation to the Hedera council's
      support contact.

### Automated Infrastructure Upgrades via Daemon

Other infrastructure components (K8s version, operator, host binaries, etc.) that need upgrading but do **not** require
a solo-provisioner provisioner self upgrade are handled by the daemon directly invoking the existing (already-installed)
solo-provisioner binary, without this self-upgrade protocol.

When such an upgrade is required, the daemon reads `manifests/infrastructure-versions.yaml` from the upgrade
package. This file specifies the desired versions for all relevant infrastructure components (e.g., Kubernetes,
container runtime, system packages). The daemon then executes the solo-provisioner binary with the appropriate
arguments, passing in the path to `infrastructure-versions.yaml`. The solo-provisioner interprets this
specification and performs the necessary upgrade actions on the host, such as upgrading Kubernetes, updating
the operator, or replacing host binaries.

This approach ensures that infrastructure upgrades are performed in a controlled and automated manner, using the same
trusted tooling already present on the host. The daemon monitors the execution and logs all actions and results for
observability and troubleshooting. If the upgrade is successful, the daemon proceeds with the rest of the upgrade
protocol; if any step fails, it records the error and halts further actions until manual intervention or a retry.

### Observability

All components produce events through three channels:

**Channel 1: JSONL Event Files** (primary, crash-safe)

Both UC and the daemon write append-only JSONL files.

**UC** uses one file per upgrade operation:

**Filename convention:**

```
<operation-type>-<YYYYMMDD>T<HHmmssZ>-<version>.jsonl
```

Example:

```
upgrade-20260415T143000Z-v0.75.0.jsonl
```

| Segment           | Example           | Purpose                                                                               |
|-------------------|-------------------|---------------------------------------------------------------------------------------|
| `operation-type`  | `upgrade`         | Type of operation                                                                     |
| `YYYYMMDDTHHmmssZ` | `20260415T143000Z` | UTC freeze timestamp — human-readable, sorts lexicographically, valid K8s label value |
| `version`         | `v0.75.0`         | Package version applied                                                               |

The UC filename is consistent with the `operationId` used in CR labels and event payloads. Each upgrade produces
exactly one file — no node-id segment is needed in the filename. The node-id is captured in every event
payload (see `nodeId` field below), which correctly records the node's identity at the time each event was
written — including any change in node-id during migration or reassignment.

**Retention policy (UC):** On startup, the UC applies a retention rule to its events directory: retain all
`upgrade-*.jsonl` files written within the last **365 days**, with a hard cap of **50 files** (sorted
lexicographically, oldest dropped first). This covers the expected cadence of at most one upgrade per month
while protecting against runaway accumulation during abnormal retry cycles.

**Daemon** uses one file per upgrade operation (same convention as UC), plus a fixed append-only file for
migration events:

Events are grouped under `daemon/events/<node-type>/<workflow>/` so each pruning algorithm operates on
exactly one directory with no pattern matching across mixed file types. Future node types (block node, relay,
etc.) each get their own subtree.

- Upgrade events (`daemon/events/consensus/upgrade/`): one file per operation.
  Filename: `consensus-upgrade-<YYYYMMDD>T<HHmmssZ>-<version>.jsonl`.
  Example: `consensus-upgrade-20260415T143000Z-v0.75.0.jsonl`.
  Retention: all files within the last **365 days**, hard cap **50 files** (oldest dropped first), applied
  on daemon startup. Same policy as UC.
- Migration events (`daemon/events/consensus/migrate/`): fixed append-only file
  `consensus-migrate-events.jsonl`. Migration soak is a long-running continuous process with no clean
  per-operation file boundary; a single fixed file fits naturally. Pruned by size (policy TBD).
  The `operationId` field in each entry identifies which migration it belongs to.

Event file locations:

- UC: `/opt/solo/weaver/uc/events/<filename>`
- Daemon (upgrade): `/opt/solo/weaver/daemon/events/consensus/upgrade/consensus-upgrade-<YYYYMMDD>T<HHmmssZ>-<version>.jsonl`
- Daemon (migration): `/opt/solo/weaver/daemon/events/consensus/migrate/consensus-migrate-events.jsonl`

volume.

### Out-of-Band Recovery for Catastrophic Post-Upgrade Failure

The recovery table above covers scenarios where the network is still running and freeze transactions can be submitted.
This section covers a distinct, lower-probability scenario: **all council nodes fail to come up after an upgrade, the
network halts, and no further freeze transactions can be delivered**. In this case the normal upgrade protocol cannot
serve as the recovery channel — the very mechanism used to deliver a fix requires a functioning network.

**Why the normal upgrade channel is unavailable.** The freeze protocol (`PREPARE_UPGRADE → FREEZE_UPGRADE`) requires
the network to be live and able to reach consensus. If the upgrade itself caused every node to fail to start, no new
transactions can be submitted and no upgrade fix can be delivered through HFS.

**Primary out-of-band recovery mechanism — public GitHub release.** A patched `solo-provisioner` binary is published
as a signed release to the public GitHub repository (`hiero-ledger/solo-provisioner`). The `hiero-ledger` organisation
is governed by the [Linux Foundation Decentralized Trust](https://www.lfdecentralizedtrust.org/) — a neutral,
multi-stakeholder foundation — making it a more trustworthy distribution point than a repository or storage bucket
controlled by a single entity. Releases are signed per the repository's published signing policy; node operators
should verify the signature before executing the binary. Each node operator independently downloads the patched binary
and runs `solo-provisioner consensus node upgrade-recover` (see Recovery Model above) on their node.

Node operators are notified of the patched release through existing council communication channels (out of scope for
this protocol). This approach requires no functioning network, no centralised delivery infrastructure, and no
per-node council coordination — each operator acts autonomously using the published, reproducible artifacts.

**Extreme catastrophic scenario — council coordination.** If the patched binary alone is insufficient to restore the
network (for example, a state corruption that requires manual intervention across a quorum of nodes), all council
members coordinate out-of-band using existing governance channels. The council brings the network up collectively, then
publishes a full Root Cause Analysis (RCA) and a record of all decisions made during the recovery window.

**Why this is the right design boundary for the protocol.** The upgrade protocol is responsible for the happy path and
for node-local failure recovery. Out-of-band recovery for a total network halt is a governance and operations concern,
not a protocol concern. Encoding a recovery delivery mechanism into the upgrade protocol would require a fallback
channel — a specific server or URL — that must remain available precisely when the Hedera network itself is down. Any
such channel would be controlled by a single entity, bypassing the council's multi-stakeholder governance. Using the
public `hiero-ledger` GitHub repository, governed by the Linux Foundation, preserves multi-stakeholder oversight
without embedding a proprietary dependency into the protocol.

## Backwards Compatibility

This HIP introduces no breaking changes to the Hedera network. The changes are entirely internal to the upgrade
orchestration infrastructure of each council node. No consensus rules, public APIs, SDK interfaces, or downstream data
consumers are affected.

The following analysis addresses each breaking change criterion in turn.

### Criterion 1: Consensus Rule Changes

**Not applicable.** The freeze transaction protocol — `PREPARE_UPGRADE`, `FREEZE_UPGRADE`, and `FREEZE_ABORT` — is
preserved exactly as-is. The protobuf message definitions, transaction validation rules, and FreezeHandler logic in the
consensus node are unchanged. The same transactions that were valid before this HIP remain valid after it. No previously
invalid transaction becomes valid, and no previously valid transaction becomes invalid.

The only change touching the consensus node is the content of the upgrade archive uploaded to HFS: a ~1 KB manifest
replaces the ~100 MB JAR-based zip. The consensus node extracts and processes the archive using the same existing
`UnzipUtility` and `FreezeHandler` code paths. From the perspective of consensus rules, the archive is an opaque byte
payload whose hash is validated — the format change is transparent to the protocol.

### Criterion 2: API, SDK, or System Contract Behavior

**Not applicable.** No HAPI transaction types, fields, response codes, or error codes are added, removed, or modified
by this HIP. The freeze transactions (`FileUpdate`, `FileAppend`, `Freeze`) behave identically. SDKs do not parse or
construct upgrade packages; they only submit the transactions. No system contracts are involved in the upgrade
orchestration path. Existing callers of any Hedera API will observe no behavioral difference.

### Criterion 3: State Structures, Record Formats, or Event Schemas

**Not applicable.** This HIP introduces no changes to the consensus state, transaction record schema, event log
format, or any data structure consumed by downstream systems. Block nodes, mirror nodes, and block explorers continue
to process freeze transactions and produce identical records. The Kubernetes Custom Resources introduced by this HIP
(`NetworkUpgradePrepare`, `NetworkUpgradeFreeze`, `NetworkUpgradeExecute`, `NetworkUpgradeFreezeAbort`) are internal
to each council node's cluster and are never visible outside the node's infrastructure boundary.

### Criterion 4: Removal or Redefinition of Existing Functionality

**Not applicable.** No existing public functionality is removed or redefined. The freeze transaction protocol continues
to work as before. The change in upgrade package format (manifest instead of JARs) affects only the council members
who author and upload the package, and the node operators who run the new upgrade tooling — not any external
participants. The HFS file (`0.0.150`) continues to be used for the same purpose; only the content uploaded to it
changes.

### Operational Considerations (Non-Breaking)

While not a network-level breaking change, this HIP does require a coordinated operational transition for council
members and node operators:

- **Upgrade package format**: The `build.zip` package is a **dual-format artifact** — it bundles both JAR-based
  content (`data/apps/`, `data/lib/`) and manifest-based content (`manifests/`, `data/config/`) in a single archive.
  During the migration period, Docker Compose nodes consume the JAR portion and K8s-native nodes consume the manifest
  portion; each node ignores the content it does not need. No simultaneous cutover of all 39 nodes is required — nodes
  migrate individually and the same package serves both models throughout. Once all nodes across previewnet, testnet,
  and mainnet have migrated, JAR production is retired and subsequent packages contain only manifest-based content.
  See HIP-1494 (Dual-Format Support During Migration) and HIP-1497 for details.
- **Node infrastructure**: A council node must be migrated from the Docker Compose model to the Kubernetes-native
  model before it uses the manifest-based upgrade path. During the migration period, unmigrated nodes continue
  participating in upgrades via the JAR portion of the dual-format package — migration does not block upgrade
  participation. The migration procedure is addressed in HIP-1497 and does not require changes to any
  public-facing interface.
- **Infra version prerequisite**: Before the first manifest-based upgrade on a node, the installed versions of
  `solo-operator`, `solo-provisioner`, CRI-O, and Kubernetes must meet the minimums declared in
  `manifests/infrastructure-versions.yaml`. If they do not, the UC emits a `ManifestInvalid` event and blocks
  the upgrade until the infra stack is brought up to date manually.
- **Tooling**: Council members who use `yahcli` or equivalent tooling to upload upgrade packages will upload
  significantly fewer transactions. The current JAR-based package (~100 MB) requires ~3,200 file-append
  transactions to upload in 4 KB chunks; the manifest-based package is a few kilobytes and uploads in single
  digits of transactions. The underlying file update/append transaction mechanism is unchanged.

### Breaking Change Impact Analysis Requirement

Per the Hiero Breaking Change Impact Analysis process, a formal impact analysis is **not required** for this HIP.
None of the four breaking change criteria are met. This HIP is an infrastructure and deployment change that is
invisible to dApps, SDKs, mirror nodes, block explorers, and end users. The only parties affected by the operational
transition are council members and node operators, who coordinate the migration as part of the normal network upgrade
governance process.

## Security Implications

### UC RBAC Scoping

The UC sidecar operates with a dedicated Kubernetes ServiceAccount whose RBAC permissions are scoped to only the four
upgrade CR types (`NetworkUpgradePrepare`, `NetworkUpgradeFreeze`, `NetworkUpgradeExecute`,`NetworkUpgradeFreezeAbort`).
It has `create`, `get`, `list`, and `watch` permissions on these resources only. It cannot modify other cluster
resources, access secrets, or escalate privileges.

### Manifest and Image Integrity

The upgrade package is uploaded to HFS and its SHA-384 hash is included in the `PREPARE_UPGRADE` transaction. Every
consensus node independently validates this hash before extracting the archive, ensuring all nodes operate from the same
manifest.

Image integrity is verified through layer hash comparison. The `layerHashes` entries in
`manifests/consensus-node-components.yaml` contain expected hashes per platform. The operator verifies these hashes
against the actual image pulled from each registry before selecting one for deployment. This prevents deployment of
tampered images even if a registry is compromised.

### CR-Based Coordination Security

The UC sidecar does not communicate directly with the `solo-provisioner-daemon`. Instead, the operator mediates by
writing to the `NetworkUpgradeExecute` CR status (setting `ReadyForProvisionerDaemon`). The daemon reads this status
from the Kubernetes API using the `solo-provisioner` service account (created by solo-provisioner at install time
with `get`, `list`, `watch` on upgrade CR types and `update` on their status subresources). A compromised container
cannot send arbitrary commands to the host-level daemon — the only communication channel available to the UC is the
Kubernetes API.

The `solo-provisioner-daemon` exposes a Unix domain socket (`/opt/solo/weaver/daemon/daemon.sock`) for
communication with the `solo-provisioner` CLI running on the same host. The socket is owned by the
`weaver:weaver` service account (UID/GID 2500) with mode `0600`, restricting access to the daemon's own process
and root. No container process can reach this socket — it is outside the Kubernetes pod filesystem and not mounted
into any pod.

The socket exposes exactly three endpoints:

| Endpoint | Purpose |
|---|---|
| `GET /health` | Liveness check — returns daemon up/down status |
| `GET /migration/consensus/soak/status` | Returns current migration soak criteria state (met/unmet, time elapsed) |
| `POST /migration/consensus/soak/start` | Activates the MigrationMonitor goroutine after a node cutover completes |

> **Future endpoints** (planned, not yet implemented): `POST /upgrade/consensus/execute` and
> `GET /upgrade/consensus/status` will expose the UpgradeMonitor's execute-phase state for
> programmatic querying. These are out of scope for this HIP.

There is **no** `POST /upgrade/abort` socket endpoint. The upgrade is forward-only — there is no node-local
abort command. Upgrade cancellation is handled exclusively through the council-level protocol:

- **`FREEZE_ABORT` transaction** — the only abort path. The council sends this transaction before the
  scheduled freeze time. The CN resets its upgrade state, writes `freeze_aborted.mf`, and the UC sidecar
  handles all cleanup automatically. No operator action or daemon interaction is required.

If a node is stuck post-freeze, the operator uses `solo-provisioner consensus node upgrade-diagnose` to
assess state and `solo-provisioner consensus node upgrade-recover` to stabilise infrastructure. The upgrade
then resumes — council coordinates any window-level recovery.

**Upgrade-monitor crash recovery**: The UpgradeMonitor goroutine does not need a state file. On daemon
restart, the goroutine starts fresh and its Kubernetes watch naturally detects the Execute CR if it is still
in `ReadyForProvisionerDaemon` state — etcd persists across daemon restarts on the hostPath volume. Each
execute-phase step is idempotent (config CR creation checks for existing CRs with a matching
`hedera.com/operation-id` label before creating new ones), so resuming from `ReadyForProvisionerDaemon` is
always safe. The soak-watcher is the only goroutine that requires explicit resume logic — it reads
`cutover-state.jsonl` to recalculate elapsed soak time rather than relying on process uptime.

Configuration files are **never embedded in any `NetworkUpgrade*` CR**. ConsensusConfig files are delivered
via dedicated Kubernetes CRs that the daemon creates during the execute phase by scanning `data/config/` in
the extracted package. Each dedicated CR kind's reconciler owns domain-specific validation and ConfigMap
management. InfraConfig files are placed directly on the host filesystem by the daemon and never enter the
Kubernetes API. This keeps all `NetworkUpgrade*` CRs well below the 1.5 MB etcd size limit regardless of the
number or size of configuration files in the upgrade package.

Dedicated ConsensusConfig CRs are never deleted — they are part of the immutable upgrade audit trail. The
daemon is idempotent: on restart it checks for an existing config CR with a matching `hedera.com/operation-id`
label before creating a new one, ensuring re-runs produce no duplicate CRs.

After a cluster rebuild, the daemon retries Kubernetes API connections with exponential backoff until the operator has
provisioned the required RBAC resources, then resumes normal processing. The daemon writes the `PendingNodeUpgrade`
status back to the cluster using a dedicated service account with limited write access to upgrade CR status
sub-resources only. Because the Kubernetes etcd data persists across cluster rebuilds (stored on a hostPath volume), the
daemon never needs to reconstruct or re-submit CRs — it only writes a single status transition.

### Solo-Provisioner Privileges

The solo-provisioner CLI requires root access (`sudo`) for operations that cannot be performed as an unprivileged
user: installing and configuring CRI-O and kubelet, managing systemd service units, and performing cluster
teardown/reinstall during infra upgrades. It creates a sandboxed environment under `/opt/solo/weaver/` with a dedicated
service
account (`weaver:weaver`, UID/GID 2500) for runtime operations. The Kubernetes cluster data, container runtime, and all
runtime state are isolated within `/opt/solo/weaver/sandbox/`.

**Daemon RBAC provisioning**: `solo-provisioner provisioner daemon install` creates the Kubernetes RBAC resources
required by the daemon: (a) a `ServiceAccount` named `solo-provisioner` in the operator namespace, (b) a `ClusterRole`
granting `get`, `list`, and `watch` on the four upgrade CR types plus `update` on their status subresources, and (c) a
`ClusterRoleBinding` linking the service account to that role. It also writes the daemon's scoped kubeconfig,
`daemon.yaml` config file, and systemd service file, then starts the daemon. This is a dedicated command separate from
`solo-provisioner operator cluster install`, which only installs solo-operator. The daemon lifecycle after initial
install is owned by `provisioner self upgrade` (stop/start via systemd); `provisioner daemon uninstall` reverses the
install. `provisioner daemon check` queries `GET /health` on the daemon socket and reports status.

**Kubeconfig after cluster rebuild**: After completing a cluster rebuild, solo-provisioner regenerates the daemon's
scoped kubeconfig and writes it to the fixed path `/opt/solo/weaver/sandbox/etc/weaver/kubeconfig`. The daemon reads
from this fixed path on every Kubernetes API call, so it automatically picks up the refreshed credential without
restart. This is a separate, least-privilege credential distinct from the CLI's cluster-admin kubeconfig at
`~/.kube/config`; no manual kubeconfig intervention is required.

### External File Download Security

Files declared in `manifests/external-files.yaml` are downloaded by the daemon before or during the freeze window.
Two controls prevent tampered or malformed files from reaching the host:

- **Checksum verification**: After download, the daemon computes the file's hash using the declared `algorithm` and compares it against the `checksum` field in `external-files.yaml`. A mismatch emits a `FileHashMismatch` event — the file is never installed. For required entries (default), the upgrade halts; for `optional: true` entries, a `FileSkipped` event is emitted and the upgrade continues.
- **Directory marker restriction**: The `destination` field must use a recognised marker prefix (`HAPIAPP_DIR`,
  `SOLO_PROVISIONER_DIR`). Arbitrary filesystem paths are rejected, preventing a malformed or compromised package
  from placing files outside the expected host locations.

### CR Audit Trail

All upgrade CRs and dedicated ConsensusConfig CRs are immutable once created and inspectable at any time via standard
Kubernetes tooling (`kubectl get`, `kubectl describe`). ConsensusConfig CRs are permanent — they represent declarative
configuration state. `NetworkUpgrade*` CRs are retained indefinitely in the current implementation; TTL-based cleanup
of terminal-phase CRs (`Succeeded`, `Failed`) by solo-operator is the intended future solution as networks accumulate
upgrades over time. The JSONL event logs written by the daemon are the durable audit record — CR deletion does not
require prior archiving.

## Benefits of the New Deployment & Upgrade Process

In summary below are the key benefits of the new Kubernetes-based deployment and upgrade process compared to the
previous Docker Compose model:

### 1. Drastically Smaller Upgrade Package

The ~100 MB JAR-based zip is replaced by a ~1 KB manifest that references pre-built container images. What previously
required thousands of HFS transactions over ~10 minutes now completes in seconds with a handful of transactions,
eliminating significant upload cost, time, and reliability risk.

### 2. Cryptographic Verification by Every Council Member

Any council member can build the consensus node image themselves from the published deterministic Dockerfile, compare
layer hashes against the `layerHashes` in the manifest, and confirm the network is running exactly the
software they reviewed. No trust in a single build authority is required.

### 3. Registry Redundancy — No Single Point of Failure

Up to 39 council members can each host a verified copy of the image in their own registry. The operator validates
all listed registries at prepare time, then **randomly selects one** per component for deployment. Random selection
serves two purposes: (1) it distributes pull load across the registry fleet so no single registry is overloaded
during a network-wide upgrade, and (2) it staggers which registry each node uses, reducing the probability that a
single registry failure causes all 39 nodes to fail simultaneously.

Node operators who have a preference — for example, preferring their own registry mirror or a geographically closer
one — can supply a **selection policy ConfigMap** that the operator reads at prepare time. If a preferred registry
passes validation, the operator selects it; otherwise it falls back to random selection from the remaining valid set.
This is an open design item — see Open Issues.

### 4. Infrastructure and Application Upgrades in One Window

A single upgrade manifest can trigger both the consensus node software upgrade and any necessary infrastructure upgrades
(Kubernetes version, operator version, host binaries) — coordinated automatically by the solo-provisioner daemon within
the same freeze window, with no separate out-of-band maintenance process required.

### 5. Full Observability with Standard Kubernetes Tooling

Every phase of the upgrade is recorded as an immutable Kubernetes Custom Resource. Operators can monitor progress with
`kubectl get`/`kubectl describe`, stream structured logs and metrics via OpenTelemetry to any compatible backend, and query the complete history of every
upgrade the node has undergone — all without custom monitoring infrastructure.

### 6. Crash-Safe and Resumable at Every Step

The system is designed to survive failures at any point: UC persists CRs to disk before sending them, etcd data is
stored on a hostPath that survives cluster teardown, and the daemon reads CR status on restart and resumes from wherever
it left off. No upgrade step requires human intervention to restart after an unexpected crash.

### 7. Clear Recovery Path When Things Go Wrong

Every failure mode emits structured events and updates the CR status with a specific reason code. Operators have
explicit commands (`solo-provisioner consensus node upgrade-recover`, `solo-provisioner consensus node upgrade-diagnose`)
for recovery scenarios, along with diagnostic artifacts (JSONL logs, a state file) that explain exactly what happened
and what action to take. The upgrade is forward-only — there is no node-local abort command.

### 8. Minimal Changes to the Consensus Node

The existing freeze transaction protocol (`PREPARE_UPGRADE → FREEZE_UPGRADE → FREEZE_ABORT`) and marker file interface
are preserved unchanged. The consensus node requires no protocol changes, eliminating the risk of introducing
regressions in consensus behaviour during the migration to Kubernetes.

## How to Teach This

This HIP changes the upgrade workflow for node operators and council members. The following guidance covers what each
role needs to understand.

### For Council Members

The core operation — submitting a `PREPARE_UPGRADE` freeze transaction — is unchanged. The key difference is what you
upload to HFS: instead of a 100 MB JAR zip, you upload a small upgrade manifest (~1 KB) that references OCI container
image digests. Your responsibilities:

1. Build or verify the consensus node OCI image against the published Dockerfile and confirm layer hashes match the
   `layerHashes` in the manifest.
2. Host the verified image in your own container registry and include your registry URL in the manifest's
   `registries` list in `consensus-node-components.yaml`.
3. Submit the manifest file ID in the `PREPARE_UPGRADE` freeze transaction as before.

Everything after submission is automated — the UC sidecar and solo-operator handle the rest.

### For Node Operators

After migration (HIP-1497), node operations change as follows:

- **Monitoring upgrades**: Use
  `kubectl get networkupgradeprepare,networkupgradefreeze,networkupgradeexecute -n <namespace>` to inspect upgrade phase
  status. Each CR shows `Pending`, `Running`, `Succeeded`, or `Failed` with structured conditions.
- **When an upgrade stalls**: Check the UC sidecar logs (`kubectl logs <cn-pod> -c uc`) and the JSONL event log at
  `/opt/solo/weaver/data/upgrade/events/`. If the daemon is stuck, use `solo-provisioner consensus node upgrade-diagnose`
  for guided diagnostics, or `solo-provisioner consensus node upgrade-recover` after an infrastructure upgrade failure.
- **Upgrade is forward-only**: there is no node-local abort command. If the upgrade cannot proceed, council coordinates
  via `FREEZE_ABORT` (before freeze) or the next upgrade window (after freeze).
- **Audit trail**: Every upgrade is recorded as an immutable set of CRs. Use
  `kubectl get networkupgradeexecute -n <namespace> --show-labels` to list all historical upgrades.

## Reference Implementation

- `hashgraph/solo-operator` — UC sidecar (`cmd/uc/`), four reconcilers (`internal/controller/`), CR API types (
  `api/v1alpha1/`)
- `hashgraph/solo-provisioner` (solo-weaver) — `solo-provisioner-daemon` (`cmd/provisioner/`),
  `provisioner self upgrade`, `consensus node upgrade-recover/upgrade-diagnose` commands
- `hiero-ledger/hiero-consensus-node` — minimal release process changes to produce OCI-manifest-based upgrade packages

## Rejected Ideas

**Using a CDN or external storage instead of HFS for the upgrade package.** While this would solve the upload size
problem, it would require changes to the freeze transaction protocol (which references an HFS FileID) and
introduce a dependency on external, centralized controlled infrastructure for a security-critical operation. The
manifest-based approach achieves the same size reduction while preserving the existing HFS-based protocol.

**Using Helm charts for upgrade orchestration instead of a custom operator with CRs.** Helm is designed for initial
deployment and version management of application stacks, not for orchestrating multi-phase stateful workflows like the
freeze-upgrade protocol. The upgrade lifecycle requires conditional logic (e.g., registry validation, two-path handling
for infra vs. CN-only upgrades), status tracking across phases, and integration with external signals (marker files). A
custom operator with purpose-built CRs provides the necessary control.

**Eliminating marker files entirely and using a pure Kubernetes-native signaling mechanism.** This would require
significant changes to the consensus node codebase, adding Kubernetes API awareness to the most security-critical
component in the network. The marker file interface keeps the CN agnostic to its deployment model, which is essential
for backward compatibility and minimizing risk.

**Having the CN itself communicate with Kubernetes (instead of UC sidecar as intermediary).** This was rejected for the
same reason as above — the CN should remain deployment-model-agnostic. The UC sidecar serves as a clean translation
layer between the CN's file-based signaling and the Kubernetes resource model.

**Dedicated infra-only upgrade window (two-phase upgrade).** An earlier design required a separate freeze
window purely for infrastructure upgrades whenever a new config file type was introduced — the operator had to
understand the new type before the CN upgrade package could be processed at prepare time. This was rejected
because it imposed unnecessary council coordination overhead (two freeze windows instead of one) and added
operational risk by requiring a network freeze solely to upgrade tooling. The dedicated-CR-per-file design
eliminates the prepare-time dependency: config files are delivered via CRs created at execute time, and the
infra components (daemon/UC) are updated with new filename→CR-kind mappings as part of the infra software
release. All upgrades — infra and CN — complete in a single freeze window, provided the required infra version
is already deployed.

The preferred long-term direction is a **rolling infrastructure upgrade** that requires no network freeze at
all: infra components (operator, daemon, UC) are upgraded node-by-node without halting consensus, similar to a
rolling Kubernetes deployment. This HIP does not specify that mechanism and is left as future work.

**Using an S3 bucket, CDN, or other centralized store as a secondary recovery delivery channel for catastrophic
post-upgrade failures.** If the network halts after an upgrade and the normal freeze protocol cannot be used, the
natural instinct is to provide a centralized channel through which a fix can be pushed to all nodes. This was rejected
for the same reason as the CDN-for-upgrade-packages alternative above: it introduces a centralized, operator-controlled
dependency into a security-critical recovery path. Any centralized store can be compromised, taken down, or
geo-blocked. The public GitHub release channel (`hashgraph/solo-weaver`, transitioning to
`hiero-ledger/solo-provisioner`) is the
appropriate mechanism — it is the established trusted source for solo-provisioner artifacts, releases are signed, and
node operators already vet this source for initial provisioning. Crucially, `hiero-ledger` is governed by the Linux
Foundation Decentralized Trust, providing multi-stakeholder oversight rather than single-entity control. Node operators
act autonomously using the signed binary; no centralized push infrastructure is
required.

## Future Improvements

### Rolling Upgrades

The upgrade model defined in this HIP applies all changes within a single network freeze window, halting
all nodes simultaneously. Two classes of rolling upgrade are worth distinguishing:

- **Rolling infrastructure upgrades** — infrastructure components (Kubernetes, CRI-O, host binaries,
  `solo-operator`, `solo-provisioner-daemon`) are upgraded node by node while consensus continues on the
  remaining nodes, with no network-wide freeze required. This is achievable without consensus protocol
  changes and is the nearer-term opportunity.

- **Rolling CN application upgrades** — consensus nodes themselves are upgraded sequentially without a
  global freeze. This requires the consensus protocol to tolerate nodes running mixed software versions
  transiently, which is a significant protocol-level change and a longer-term consideration.

Neither mechanism is designed in this HIP. A dedicated follow-on HIP will address whichever variant
becomes feasible first.

### Zero-Downtime CN Upgrade via Synchronized Dual-Stack Cutover

A third upgrade model — distinct from both rolling variants above — is currently under active design. Unlike
rolling upgrades (which upgrade nodes sequentially) or the freeze model (which halts all nodes simultaneously),
this approach achieves zero downtime through a **network-wide synchronized cutover at a predetermined block
boundary**:

- **Two software stacks run in parallel** on each node simultaneously. The V+1 process is started and fully
  warmed up — loading state and reaching readiness — before any cutover occurs. It does not participate in
  consensus or generate blocks during the warm-up phase.
- **A predetermined block number N is agreed across all nodes.** Each node continues generating blocks
  normally using V up to and including block N.
- **At block N+1, all nodes atomically switch to V+1.** Because V+1 is already warmed up and the cutover
  block is known in advance, the handover is near-instantaneous and network-wide — preserving the global
  consistency guarantee of the current freeze model without incurring downtime.
- **After the cutover, V is wound down.** Once V+1 is confirmed producing blocks, the old process is
  gracefully terminated and its resources released.

This model requires the CN to support running two software stacks in parallel, a mechanism for agreeing on
the cutover block number across all council nodes, and a warm-up readiness signal that the orchestration
layer can observe before committing to the switch. The design of these mechanisms is in progress and will
be specified in a dedicated follow-on HIP.
